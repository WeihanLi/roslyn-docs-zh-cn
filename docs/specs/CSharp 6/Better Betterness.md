# 更优的"更优性"（Better Betterness）

"Better Betterness"是 C# 6 中的一项语言变更，存在于随 Visual Studio 2015 及后续版本发布的 Roslyn C# 编译器中。无论命令行上指定了何种语言版本，编译器均会应用此变更。因此，即使使用 `/langver:5` 将编译器作为 C# 5 编译器使用，这些新规则也会生效。

以下是修改后语言规范的相关章节：

### 7.5.3.3 表达式的更优转换

给定一个从表达式 E 到类型 T<sub>1</sub> 的隐式转换 C<sub>1</sub>，以及一个从表达式 E 到类型 T<sub>2</sub> 的隐式转换 C<sub>2</sub>，若 E 与 T<sub>2</sub> 不完全匹配，且以下条件之一成立，则 C<sub>1</sub> 是比 C<sub>2</sub> ***更优的转换***：
- E 与 T<sub>1</sub> 完全匹配（§7.5.3.4）
- T<sub>1</sub> 是比 T<sub>2</sub> 更优的转换目标（§7.5.3.5）

### 7.5.3.4 完全匹配的表达式

给定表达式 E 和类型 T，E ***完全匹配*** T 指以下条件之一成立：
- E 具有类型 S，且存在从 S 到 T 的恒等转换
- E 是匿名函数，T 是委托类型 D 或表达式树类型 Expression<D>，且以下条件之一成立：
  - 存在针对 D 的参数列表上下文中 E 的推断返回类型 X（§7.5.2.12），且存在从 X 到 D 的返回类型的恒等转换
  - E 是非异步的且 D 的返回类型为 Y，或者 E 是异步的且 D 的返回类型为 Task<Y>，且以下条件之一成立：
    - E 的主体是与 Y 完全匹配的表达式
    - E 的主体是语句块，且每条 return 语句均返回与 Y 完全匹配的表达式

### 7.5.3.5 更优的转换目标

给定两种不同类型 T<sub>1</sub> 和 T<sub>2</sub>，若以下条件之一成立，则 T<sub>1</sub> 是比 T<sub>2</sub> ***更优的转换目标***：
- 存在从 T<sub>1</sub> 到 T<sub>2</sub> 的隐式转换
- T<sub>1</sub> 是委托类型 D<sub>1</sub> 或表达式树类型 Expression&lt;D<sub>1</sub>>，T<sub>2</sub> 是委托类型 D<sub>2</sub> 或表达式树类型 Expression&lt;D<sub>2</sub>>，D<sub>1</sub> 的返回类型为 S<sub>1</sub>，且以下条件之一成立：
  - D<sub>2</sub> 返回 void
  - D<sub>2</sub> 的返回类型为 S<sub>2</sub>，且 S<sub>1</sub> 是比 S<sub>2</sub> 更优的转换目标
- T<sub>1</sub> 是 Task&lt;S<sub>1</sub>>，T<sub>2</sub> 是 Task&lt;S<sub>2</sub>>，且 S<sub>1</sub> 是比 S<sub>2</sub> 更优的转换目标
- T<sub>1</sub> 是 S<sub>1</sub> 或 S<sub>1</sub>?（其中 S<sub>1</sub> 是有符号整型），且 T<sub>2</sub> 是 S<sub>2</sub> 或 S<sub>2</sub>?（其中 S<sub>2</sub> 是无符号整型）。具体而言：
  - S<sub>1</sub> 为 `sbyte`，S<sub>2</sub> 为 `byte`、`ushort`、`uint` 或 `ulong`
  - S<sub>1</sub> 为 `short`，S<sub>2</sub> 为 `ushort`、`uint` 或 `ulong`
  - S<sub>1</sub> 为 `int`，S<sub>2</sub> 为 `uint` 或 `ulong`
  - S<sub>1</sub> 为 `long`，S<sub>2</sub> 为 `ulong`

##### 说明

"从类型的更优转换"这一概念已不再适用。

请注意，新规则去除了大量基于表达式形状的特殊情况处理。在新规则下，表达式的意义仅在于其是否与类型"完全匹配"——完全匹配始终具有最高优先级。（这保留了以下性质：你始终可以通过将参数转型为该重载所期望的精确类型来选择特定重载。）

"完全匹配"的复杂性主要来自于"透视"lambda 并匹配返回类型的逻辑。旧规则允许透视恰好一层 lambda（这似乎完全是任意的），而现在已推广为任意层数：

- `7` 与 `int` 完全匹配
- `() => 7` 与 `Func<int>` 完全匹配
- `() => () => 7` 与 `Func<Func<int>>` 完全匹配，但在早期版本中并非如此！！！

"透视"委托和任务的规则现在不再依赖参数表达式的形状（旧规则要求是 lambda 和 async 形式），因此已被纳入"更优的转换目标"，该规则忽略表达式本身。关于可空类型的规则此前一直存在缺失，现已补全——部分内容在旧编译器中已"意外"实现，大部分为新行为。

##### 兼容性修复

在 VS2015 Update 2（Roslyn 1.2 版本）中，我们发现并修复了变更后的重载解析规则中的一处细微不兼容问题。受影响代码的示例参见 https://github.com/dotnet/roslyn/issues/6560。为修复此问题，§7.5.3.5 第二条规则修改如下：

> 对于相应参数的方法组转换（§6.6），如果更优的转换目标（§7.5.3.5 更优的转换目标）是委托类型，但该委托类型与方法组中唯一最优方法（§6.6 方法组转换）不兼容（§15.2 委托兼容性），则两种委托类型均不更优。

此解释不可避免地有些非正式（例如，§7.5.3.5 中没有"相应参数"的概念）；将其纳入规范需要对规范进行重组。

供参考：

### 15.2 委托兼容性

方法或委托 M 与委托类型 D 兼容，当且仅当以下所有条件均成立：
- D 和 M 具有相同数量的参数，且 D 中每个参数与 M 中对应参数具有相同的 ref 或 out 修饰符。
- 对于每个值参数（无 ref 或 out 修饰符的参数），存在从 D 中参数类型到 M 中对应参数类型的恒等转换（§6.1.1）或隐式引用转换（§6.1.6）。
- 对于每个 ref 或 out 参数，D 中参数类型与 M 中参数类型相同。
- 存在从 M 的返回类型到 D 的返回类型的恒等或隐式引用转换。
