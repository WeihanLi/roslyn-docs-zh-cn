.NET 与 Terrapin 流程
===
Terrapin 系统负责验证 .NET 生态系统中 NuPkg 的来源。具体而言，它需要验证 NuPkg 中的 DLL 可以使用公开可用的构件重新构建。

本文档描述了 Terrapin 用于验证 .NET 二进制文件来源的工作流程、各团队的职责，以及 .NET 团队将交付给 Terrapin 团队以支持服务成功运行的工具。

## .NET 工具
.NET 团队将向 Terrapin 团队交付一套工具，提供必要的验证功能。之所以提供一系列工具而非一个一体化工具，是为了更好地分离不同的职责，并让相应团队来承担各自的责任：

1. 提供验证所需构件列表（由 .NET 负责）
1. 获取验证所需构件（由 Terrapin 负责）
1. 执行验证并判定来源（由 .NET 负责）

未来我们可能会提供一个"一体化"的工具体验，但短期内我们将专注于保持这些职责和工具的独立性。

### dotnet-build-manifest-generator
这是一个全局工具，接收 DLL/PDB 文件组合，并生成一个构件清单文件，列出执行验证所需的全部构件。其中还包括执行验证所需使用的 `dotnet-build-validator` 版本。

该工具旨在与旧版 .NET 编译器生成的二进制文件配合使用。每当 PDB 来源元数据的内容发生变化时，都会生成新版本的工具以适应新的元数据格式。Terrapin 应始终在其服务中安装最新版本的此工具。

### dotnet-build-validator
这是一个全局工具，接收以下参数：待验证的 PE 文件、对应的 PDB 文件，以及一个包含从构件清单文件中获取的构件的目录。该工具随后会返回所提供的 PDB 文件是否能够通过来源验证。

该工具有三种输出状态：

1. 验证成功
1. 验证失败，因为二进制文件不匹配
1. 验证失败，因为无法生成二进制文件

该工具将与编译器同步发布：每次有新编译器时，都会有一个使用该编译器二进制文件的新版本工具。这一点非常重要，因为编译器不保证不同版本之间的确定性输出。即使是编译器的小版本差异，如果代码涉及 bug 修复或优化，也可能为相同源代码生成不同的 IL。为使验证器以高保真度运行，Terrapin 必须使用与编译器同步发布的版本。

Terrapin 应将此工具作为本地工具安装用于特定验证事件。这样每次验证事件都可以使用不同版本的 `dotnet-build-validator`，而不必担心全局安装时可能出现的版本冲突问题。

## Terrapin 服务工作流程
Terrapin 服务将对 NuPkg 文件中的每个 DLL 执行以下工作流程进行验证：

1. 在 DLL + PDB 组合上执行 `dotnet-build-manifest-generator`。
1. 为验证设置环境：
    1. 切换到清单中列出的对应操作系统。后续所有操作预期在此机器上运行。
    1. 在目标机器上安装正确的 .NET 运行时（如有必要）。
1. 下载构件清单文件中列出的构件，以及指定版本的 `dotnet-build-validator`。
1. 执行 `dotnet-build-validator`，提供 DLL、PDB 以及构件下载目录。

Terrapin 服务将把构件下载到由 Terrapin 创建的目录中。该目录将包含两个子目录，分别用于存储源文件和引用文件：`sources` 和 `references`。这些目录中构件的命名将在清单文件中指定。这样就将避免路径冲突和文件路径修正的责任落在编译器团队身上。

## 构件清单文件
`dotnet-build-manifest-generator` 的目标是向 Terrapin 服务提供一个清单文件，列出完成验证所需的全部构件以及运行验证所需的环境。所需构件包括：

1. 编译所用的源文件。
1. 编译所用的引用 DLL。

目前的预期是，Terrapin 需要重建大部分用于创建二进制文件的环境才能执行来源验证。这包括匹配的 .NET 运行时版本和操作系统。这是因为 OS 依赖项（如 zip 压缩算法版本）以及操作系统版本，都可能影响编译器的确定性输出。我们在 Terrapin 中的实验可能会促使我们重新评估这些因素对确定性的影响，但目前的设计是在我们现有假设的基础上构建验证。

为此，构件清单文件将包含所有必要的环境信息，包括操作系统 RID、.NET 运行时版本，以及运行编译器时使用的是 Desktop 还是 .NET Core 运行时。

构件清单文件将是如下格式的 JSON 文件：

```json
{
    "sources": [
        { 
            "uri": "https://github.com/dotnet/roslyn/blob/f5517311e37a3883e09dc6daa5069bbc4f44c882/src/Compilers/Shared/Csc.cs",
            "destFileName": "c6fea380dac7392b945fa6c63466746f830c5cc3"
        }
    ],
    "references": [
        {
            "name": "System.Reflection.Metadata.dll",
            "mvid": "51ef3ce0-5a89-4325-b094-8a5970acc676",
            "destFileName": "51ef3ce0-5a89-4325-b094-8a5970acc676",
        }
    ],
    "environment": {
        "compiler-language": "csharp",
        "compiler-version": "3.8.0",
        "os": "windows",
        "os-version": "...",
        "dotnet-runtime": "net5.0.0"
    }
}
```

## 验证阶段
围绕这项工作仍有许多待解决的问题，例如代码页和行尾的影响，以及 .NET 运行时版本对输出确定性的影响程度。回答这些问题的最佳方式是将真实客户代码推送到服务中，以验证这些问题的实际影响。此外，我们还需要解决一些常规流程问题，例如交付节奏，以及确保各个部分能够顺畅衔接。

为了同时解决这两个问题，我们将把这项工作分解为一系列复杂度递增的里程碑，逐步接近我们对生态系统中所有主要 NuPkg 进行来源验证的目标：

里程碑如下：

- Roslyn 能够在刚构建完 dotnet/roslyn 的机器上成功验证 dotnet/roslyn。这将使我们确信底层的来源检查是可行的。执行验证所需的所有构件都将位于机器上的已知位置，从而可以将"构件获取"这一环节单独分离出来。
- Roslyn 将为 Terrapin 生产 dotnet-build-manifest-generator 和 dotnet-build-validator，Terrapin 将验证他们能够在本地托管的项目上使用这些工具。
- 在 Windows 上重新构建 DLL：此里程碑的目标是使用 .NET Framework 编译器在 Windows 上生成的 NuPkg。目标是能够重新构建 DLL，无论验证是否成功。这将帮助我们确保 Terrapin 与 .NET 之间的流程正确，并验证 PDB 中确实包含足够的信息以重现编译。
- 在 Windows 上验证 DLL：此里程碑的目标是使用 .NET Framework 编译器在 Windows 上生成的 NuPkg。目标是验证这些二进制文件的来源，并找出来源验证失败的常见原因。这将使我们开始与 .NET 客户讨论如何更改其构建以使其适合来源验证。
- 在 Windows 上验证 DLL（第 2 部分）：此里程碑的目标是使用 .NET Core 编译器在 Windows 上生成的 NuPkg。这将要求 Terrapin 额外完成安装正确 .NET Core 运行时并使用其驱动安装流程的环境设置步骤。
- 验证在 Linux/Mac 上生成的二进制文件：此里程碑的目标是在 Linux/Mac 上生成的 NuPkg。这将要求 Terrapin 在验证流程中引入多台机器，同时帮助我们了解 OS 子版本对来源验证能力的实际影响。

### 常见问题

### VS 16.7 发布之前生成的二进制文件怎么处理？
来源验证流程依赖于 PDB 中包含的额外元数据，编译器使用这些元数据生成构件清单文件并重建编译。这些额外信息首次添加到 PDB 是在 Visual Studio 16.7 版本中。这意味着在此之前生成的任何二进制文件都将无法通过来源验证。

这是一个已知问题，此类旧版 NuPkg 不会被支持来源验证。

### 如何处理源代码生成器？
来源验证将使用源代码生成器的输出，而非在验证过程中重新运行源代码生成器。这将简化流程，同时消除源代码生成器（客户自定义工具，不具备我们的严格程度）在确定性方面的问题。

## 待解决问题

### 在 .NET Framework 上运行验证器
对于 Windows，预计我们应该能够使用 .NET Core 来运行针对 .NET Framework 编译器生成的二进制文件的验证器。我们在 Unix 操作系统上担心的运行时差异问题在 Windows 上并不存在。因此，任何能够执行编译器的 .NET 运行时都应该能够进行来源验证。

不过，在实际使用中我们可能会发现差异。届时我们将需要为 `dotnet-build-validator` 提供一种模式，使其能够调用 .NET Framework 进程来执行验证。

### 行尾问题
为了验证成功，`dotnet-build-validator` 接收的源文件必须与提供给原始编译的源文件具有相同的行尾。这会影响 PDB 中存储的源文件校验和、字符串字面量的内容等。

Source Link 提供的信息可能不足以正确重建具有正确行尾的文件。例如，`.gitattributes` 文件可能会改变不同操作系统上行尾的使用方式。

这是调试器团队今天遇到的问题，他们已通过一个[简单算法](https://github.com/dotnet/sourcelink/pull/678/files#diff-3339ce7022ba5de3029d3f15bdf8ca585905bad4f25077ca1b1616c31d841c62R258-R290)解决了这个问题，我们可以借鉴：本质上，如果文件具有一致的 LF 或 CRLF 行尾，且 SHA 不匹配，则翻转行尾；如果翻转后匹配，则使用该版本。

### 未来扩展
本文档描述的是 C# 和 VB 编译器的验证流程，但这些并非 .NET SDK 中创建或修改 DLL 的唯一工具。其他工具包括但不限于 F# 编译器和 IL 裁剪工具。

预计这些工具可以以最小的变更纳入本文档所述的流程。例如，构件清单文件可以扩展以包含新的构件类型，Terrapin 可以使用新工具来验证构建的各个阶段。
