### 笔记

#### 源代码生成器以外的内容：

* `Project.RemoveDocuments`：此类型中的所有其他方法均使用 `IEnumerable<T>` 而非 `ImmutableArray<T>`。如果能从头重来，我们可能会统一使用后者，但鉴于现有方法均使用前者，此处也应保持一致。这同样适用于 `Solution` 上的重载。
* `Solution.RemoveAdditonalDocuments/RemoveAnalyzerConfigDocuments`：我们已将这些方法添加到 `Solution`，但没有添加到 `Project`，尽管后者也有各自的单独 `Remove` 方法。应将这些方法也添加到 `Project`。
* `DocumentOptionSet.GetOption`：此 API 作为具体类 `DocumentOptionSet` 上的 `new` 方法添加，以应对我们将实现移至 `OptionSet` 并将该方法改为非虚方法时的变化。这实际上并不是一个破坏性更改，因此应将其移除。

#### 源代码生成器：

总体说明：此处所做的更改在 16.6 发布后**不**被视为破坏性更改。
不过，我们对 API 表面进行了整体审查，并为 @chsienki 提供了一些意见。

* `GeneratorAttribute`：应考虑为此类型添加 `LanguageName` 属性。
* `InitializationContext`：重命名为类似 `SourceGeneratorInitializationContext` 的名称。当前名称存在歧义。
* `InitializationContext.RegisterForNotifications`：
    * 命名与分析器注册方法现有的 `Action` 后缀命名规范不一致。
    * `SyntaxReceiverCreator`：分析器中所有其他 `RegisterForXAction` 方法均使用 `Action` 或 `Func`（视情况而定）。应考虑在此处也遵循同样的做法。
* `ISyntaxReceiver`：此接口可能与现有 API 存在冗余。
* `SourceGeneratorContext`：目前没有办法在源代码生成器之间共享语义模型和其他绑定信息。应设计一种共享此类信息的方式，以避免各生成器引入大量重复的绑定操作。
