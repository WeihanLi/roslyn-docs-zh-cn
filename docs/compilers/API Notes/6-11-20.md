### `BaseTypeSyntax` 设计

在主分支中，我们目前为 record 主构造函数的 `SimpleBaseTypeSyntax` 添加了可选的参数列表。然而，有一种观点认为我们应该添加 `BaseTypeSyntax` 的新子类型（例如 `PrimaryConstructorTypeSyntax`），用于表示这些新的基类型子句。这个扩展点在原始设计中正是为此目的而明确预留的。对于这两种设计，`GetSymbolInfo` 的行为将保持一致：如果 `BaseTypeSyntax` 有参数列表（无论是作为独立类型还是作为 `PrimaryConstructorTypeSyntax`），它将返回构造函数符号或候选符号（视情况而定）；如果没有参数，则不返回任何内容。

保留 `SimpleBaseTypeSyntax` 上可选参数列表的优点：
* 主构造函数参数感觉像是对现有节点的扩展，而非全新的节点类型。
* 现有的分析器/代码修复器可能无需太多修改即可正常工作。

扩展 `BaseTypeSyntax` 的优点：
* 这个扩展点正是为此目的而明确添加的。
* 现有代码不太可能错误地使用它。
* 有一种感觉认为，它将为此语法的未来扩展（如果此类 C# 特性将来出现）提供更清晰的分界。

#### 结论

我们倾向于在下一个预览版中扩展 `BaseTypeSyntax`。

### `CompilationOutputFilePaths`

当前这个结构体只有一个路径，复数形式显得有些奇怪。以下是一些重命名选项：

* `CompilationOutputFileInfo`
* `CompilationOutputInfo`
* `CompilationOutputFiles`

具体选择哪个名称留给执行重命名的人来决定。

### `SolutionInfo.Create`

为兼容性重载添加 `EditorBrowsable(Never)`。

### `SymbolKind.FunctionPointer`

为与其他同时也是 `TypeKind` 的种类保持一致，在后面加上 `Type` 后缀。

### `SyntaxFactory.ParseType`

新的重载在两个工厂中均应使用 `ParseOptions`，而非特定的 C#/VB 类型。此处大多数接受 `ParseOptions` 的 `Parse` 重载都使用非特定类型，这对 IDE API 非常方便。虽然类型安全性略有降低，但人机工程学和一致性方面的优势是决定性因素。

### `WithExpressionSyntax.Receiver`

为与其他 `SyntaxNode` 保持一致，应重命名为 `Expression`。

### `SymbolFinder.Find(Derived(Classes|Interfaces)|Interfaces)Async`

应重新考虑添加多个重载的方式，转而从现有重载中移除默认参数，并引入一个将 `transitive` 设为 `true` 的新重载，以匹配当前行为。这是我们处理此类场景的标准做法。

### `INegatedPatternOperation.NegatedPattern`

重命名为 `Pattern`。

### `IWithOperation.Value`

重命名为 `Operand`。

### `Renamer.RenameDocumentAsync`

除非有合作团队准备好在 16.7 中使用该 API，否则将此方法及相关结构体设为内部（internal）。待有人准备好使用并验证 API 形态后，可以再将其公开。
