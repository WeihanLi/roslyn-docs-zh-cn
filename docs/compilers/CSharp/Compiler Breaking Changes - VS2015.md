**本文档列出了 Roslyn（VS2015 及更高版本）相对于本机 C# 编译器（VS2013 及以前版本）的已知重大更改。**

### (C#) 与语言规范或本机编译器的偏差

此页面追踪 Roslyn 编译器有意偏离以前版本或语言规范的已知情况。它既包括重大更改，也包括使得以前在语言规范或本机编译器中不被允许的情况成为可能的更改。没有链接的 bug 编号指的是 Microsoft 内部 bug 跟踪系统。

- 在某些情况下，由于本机编译器中的 bug，包含一个或多个类型参数的结构的指针在没有错误的情况下编译。所有此类程序现在应该在 Roslyn 中产生错误。有关示例和详情，请参见 [#5712](https://github.com/dotnet/roslyn/issues/5712)。
- 当在带有动态参数的静态上下文中调用只有实例方法的方法组时，本机编译器没有生成警告/错误，并生成始终在执行时抛出的代码。Roslyn 在这种情况下会产生错误。
- 本机编译器曾对抽象类的未使用/未赋值字段生成警告（169、414、649）。Roslyn 1（VS 2015）不产生这些警告。有了 [#14628](https://github.com/dotnet/roslyn/pull/14628)，这些警告应该再次被报告。
- 在 VS 2017 和 C# 7.0 的预览和 RC 版本中，允许使用返回非 void 类型的 Deconstruct 方法进行解构。从 RC.3 开始，Deconstruct 方法需要返回 void。

### C# 规范偏差：逻辑运算符重载解析

（来自 DevDiv #656739）

C# 规范第 7.12 节指出，形如 `x && y` 或 `x || y` 的操作通过将操作视为 `x & y` 或 `x | y` 来处理重载解析。

在 DevDiv #656739 之前，Roslyn 报告上述示例中 || 是有歧义的，因为 | 会是这样。但不幸的是，dev11 仅在考虑用户定义的运算符（或其中一个操作数是动态的）时将 || 视为 |。也就是说，对于内置运算符，它区分 || 和 |。由于整型没有 || 运算符，因此没有歧义。

### C# 规范偏差：枚举类型的 new 表达式可以具有常量值

根据规范，枚举类型的 new 表达式不能具有常量值。但是，本机编译器对无参数值类型构造函数违反了这一规定，Roslyn 通过对无参数值类型构造函数进行常量折叠来保持兼容性。这也适用于枚举类型。

### C# 规范偏差：New Int32() 被视为常量 0

```c#
// 故意违反规范：
// 规范不允许 "new int()" 被视为常量
// 本机编译器将 "new int()"（等）视为"零"常量表达式，
// 尽管规范没有将对象创建表达式包含在合法常量表达式列表中。
```

### C# 规范偏差：无参数构造函数

见 bug 4424

``` cs
class C
{
    public C(params int[] x) { }
}
class D : C
{
}
```

**实际结果**：error CS1729: 'C' 不包含接受 0 个参数的构造函数

**预期结果**：无错误（尽管规范要求恰好是无参数构造函数，Dev10 和以前的版本始终支持可以用空参数列表调用的任何构造函数）

### C# 规范偏差：括号中的 null

根据规范，以下代码是非法的：

``` cs
Object o = (null);
```

因为 null 转换不适用于括号中的 null 表达式，只适用于 null 字面量。

``` cs
    {
        I3 p = new Program();
        MyAction<long> o = p.Y; // 查找是否有歧义？
        long l = 12;
        o(l);
    }
}
```

### C# 规范偏差：Dev10 不报告不可达的空语句（C#）

Dev10 不报告以下不可达语句，我们也不报告：

``` cs
class Program
{
    static void Main()
    {
        if (false)
        { // 此块不可达
        }
    }
}
```

更精确地说，Dev10 不会报告不可达的"空"语句、throw 语句或块语句本身。但是，如果不可达块中包含其他类型的（不可达）语句，Dev10 会报错。

### C# 规范偏差：静态类作为接口方法的参数（C#）

Dev10 允许静态类用作接口方法参数和接口方法返回类型。这是语言规范（10.1.1.3.1）不允许的，但 Dev10 编译器允许。

我们在 Roslyn 中遵循相同做法，以避免破坏（可能无用的）现有程序。

### C# 规范偏差：null??null 和其他 null 表达式

``` cs
// 注意：我们在此扩展了规范。C# 3.0 规范没有描述
// "null 类型"。相反，它说 null 字面量没有类型，并且可以
// 转换为任何引用或可空类型。但是，C# 2.0 和 3.0
// 实现有一个"null 类型"，某些表达式（而不是 null 字面量）
// 可能具有这种类型。（例如，(null??null)，这也是规范的扩展。）
```

### C# 规范偏差：括号中的 lambda 表达式

从技术上讲，在规范中，括号中的 lambda 表达式不是匿名函数表达式（因此从不合法）。但是，Dev10 和 Roslyn 都忽略括号（除了优先级），接受如下代码：

``` cs
var tmp = (Func<int,int>)( x => x+1 );
```

### 方法隐藏和重载解析

C# 语言规范说隐藏的方法在重载解析过程中会被提前删除。但这是不正确的。以下程序（在 Dev11 和 Roslyn 中工作）演示了具有相同签名的方法不被隐藏。

``` cs
using System;

class Base
{
    public void M(int x) { Console.WriteLine("Base.M(x:" + x + ")"); }
}

class Derived : Base
{
    public new void M(int y) { Console.WriteLine("Derived.M(y:" + y + ")"); }
}

public class Test
{
    public static void Main()
    {
        Derived d = new Derived();
        d.M(x: 1);
        d.M(y: 2);
    }
}
```

### 保留成员

本机编译器（不当地）将 C# 语言中保留成员的概念扩展到了类型参数。Roslyn 不拒绝此代码。

### 编译原始类型

特殊类型 Int32 的源代码是一个包含其自身类型成员的结构。这不是 C# 语言规范允许的，但编译器必须允许它，以便为该原始类型生成元数据。其他原始类型也一样。

### ?? 运算符右操作数不必在左操作数为非 null 常量时确定赋值

以下测试用例应该"通过"以匹配 Dev10 行为：

``` cs
using System;
static class Program
{
    static void Main()
    {
        const string x = "pass";
        string y;
        string z = x ?? y;
        Console.WriteLine(z);
    }
}
```

但是，当前语言规范不支持这一点。由于 Dev10 接受此代码，建议修改规范以允许这样做。

### Roslyn 比以前的编译器需要更多程序集引用

Roslyn 不会忽略引用来自编译中缺少程序集引用的符号的方法重载。如果尝试使用包含此类方法的方法组（例如在重载解析中），此类方法在以前的编译器中被忽略。在 VS2015 中，会给出错误 CS0012。参见 [#9370](https://github.com/dotnet/roslyn/issues/9370)。

### Roslyn 有时需要在访问 VB 索引属性时使用索引运算符

虽然 Roslyn C# 编译器允许省略可以不带参数调用的索引属性的索引运算符（例如，它有一个带默认值的参数），但 C# 编译器不允许对结果进行第二次索引，除非第一次索引操作是显式的。参见 [#17045](https://github.com/dotnet/roslyn/issues/17045)。
