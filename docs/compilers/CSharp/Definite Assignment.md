明确赋值
===================

C# 的明确赋值规则在 Roslyn 中重新实现了，我们发现了规范和原生编译器的一些缺陷，我们在这里记录下来。

### 常量布尔表达式

C# 语言规范和原生编译器中的明确赋值实现缺少两个基础明确赋值规则。缺少这两个规则导致编译器在大量情况下出现不直观的行为，因此原生编译器还包含大量临时规则来修补这些示例。

Roslyn 编译器实现了带有两个附加基础规则的 C# 语言规范。添加这些规则后，原生编译器实现的临时规则不再需要。新规则比原生编译器实现的规则更精确；例如，Roslyn C# 编译器接受以下代码，但原生编译器拒绝它：

```cs
    static void Main(string[] args)
    {
        int x;
        if (false && x == 3)
        {
            x = x + 1; // Dev10 不认为 x 是明确赋值的
        }
    }
```

新的基础规则是：

> #### 5.3.3.N 常量表达式
对于值为 true 的常量表达式：
 - 如果 *v* 在表达式之前是明确赋值的，
   则 *v* 在表达式之后是明确赋值的。
 - 否则 *v* 在表达式之后是"在 false 表达式之后明确赋值的"。

>对于值为 false 的常量表达式：
 - 如果 *v* 在表达式之前是明确赋值的，
   则 *v* 在表达式之后是明确赋值的。
 - 否则 *v* 在表达式之后是"在 true 表达式之后明确赋值的"。

>对于所有其他常量表达式，*v* 在表达式之后的明确赋值状态与 *v* 在表达式之前的明确赋值状态相同。

### 控制转移和中间 finally 块

C# 语言规范中的明确赋值和可达性规范没有正确考虑控制转移源和目标之间"中间"finally 块的可能性。这些 finally 块可以赋值给变量，从而改变明确赋值状态，或改变控制转移行为（例如，通过抛出异常）。原生编译器和 Roslyn C# 编译器在计算明确赋值和语句可达性时都考虑了它们。

### 跨程序集的结构明确赋值

编译器以前有一个错误，它不认为结构内部引用类型的私有字段参与该结构类型变量的分段明确赋值。例如，

```C#
public struct S
{
    private object _f;
}

---

public class C
{
   void M()
   {
      S s; // s 现在被认为是明确赋值的
   }
}
```

在前面的示例中，如果 `S` 和 `C` 在同一程序集中，`s` 将被认为不是明确赋值的，但如果它们在不同的程序集中，`s` 将被认为是明确赋值的。这种行为在较新的编译器中为了向后兼容而保留，但它不符合任何版本的 C# 语言规范。
