 
**与以前编译器相比的代码生成差异。**

**注意** - **编译器生成的 IL 流的形式和形状不是公共契约！**
编译器只负责生成与 C# 源代码语义等效的 IL。
对输出精确形式的任何依赖可能会被新版本甚至错误修复所破坏。

这可能不全面，因为更改是一个移动的目标，可能有一些不太明显或重要的更改没有提及。本文档始终在进行中...

**本文档的目的是什么？**
* JIT 编译器通常努力处理任何有效的 IL。然而，现实是大多数优化和大多数测试都针对"典型"代码生成，因此对编译器输出的任何更改都有兴趣，以便 JIT 可以确保有足够的支持。
* 有一类工具依赖于反编译技术。通常这类工具*选择*通过专门针对特定编译器输出而被破坏。适应新模式只是他们作为生命周期的一部分必须处理的事情。
* 对这些更改也有相当多的学术兴趣。

**以下是更改的大致列表：** 
这里有两个部分 - Release 代码生成和 Debug。它们是分开的，因为更改通常性质不同，因为它们由不同的目标驱动。
通常 Release 代码生成努力成为源代码语义最高效和紧凑的表示，而 Debug 代码生成重视可调试性。
当效率和可调试性冲突时，Release 和 Debug 做出不同的选择。

**== Release（优化）**

•	async 代码生成在 Roslyn 中大部分被重新完成。
 - 当启用优化时，捕获现在更智能，因为它依赖于精确的数据流分析。基本上只有跨越 await 的局部变量需要被提升。旧编译器使用更保守的方法，经常会不必要地捕获局部变量。
 - 栈溢出完全不同。我们生成强类型可重用槽，而不是始终存在的对象数组槽，但仅在需要时才生成。

•	迭代器 - 一般原则相同，但状态机有一些小的改进。
 - 局部捕获基于精确的数据流分析，可能导致提升到迭代器类的局部变量更少。通常只有值在 yield 语句之间存活的局部变量需要被捕获。
 - 有效状态现在都是正的，无效状态是负的。迭代器主体中的常见路径在有效状态上切换，这些状态的集合是连续的有好处。
 
•	Lambda 在缓存策略上有一些小的更改，以及非提升 lambda 的表示方式发生了更改。
- 缓存策略已更改为更紧凑的模式，这也导致对缓存字段的访问更少（如果 JIT 不优化多次读取）。
而不是等效于
 
```C#
 if (cacheField == null)
 {
   cacheFiled = {allocate new lambdaDelegate};
 }
 return cacheField;
```
我们现在生成接近于

```C#
return cacheField ?? (cacheFiled = {allocate new lambdaDelegate});
```

- 非提升 Lambda 表达式现在实现为单例显示类上的实例方法。由于委托的入口点是实例 "Invoke" 方法，如果底层实现方法也是具有与 "Invoke" 完全相同的形式签名的实例方法，则在运行时将委托调用分派到该方法更便宜。
- 泛型方法中非提升 Lambda 的委托现在可以被 Roslyn 缓存。这主要是上述更改的一个受欢迎的副作用。

•	Roslyn 中的字符串 switch 代码生成是全新的。Roslyn 不使用字典以避免分配和字符串 switch 首次执行时可能出现的巨大惩罚。Roslyn 使用一个私有函数将字符串映射到哈希码和数字 switch。从某种意义上说，这是对以前使用静态字典的技术的部分内联。

•	数组初始化器稍微紧凑一些 - 在大多数情况下，避免了数组实例的额外临时变量，改用 dup。（我认为 VB 一直这样做，但这对 C# 来说是新的）

•	嵌套 try 的 leave 在许多情况下不会级联通过外部区域，而是直接离开到最外层区域。（VB 一直这样做，现在 C# 也这样做，因为这部分代码生成在共享库中，而且它也更紧凑）
分支线程通常可能比旧编译器处理更多情况。Leave-to-leave 情况可能是最明显的。

•	数字 switch – 
- 在具有非零最小键的 switch 中重新偏置和范围检查方面有一些小的更改。
- 在二进制搜索和计算 switch 桶之间选择的策略略有不同。
在高级别上，switch 代码生成遵循与旧编译器相同的模式，但我们修复了一些可能在某些情况下导致不平衡决策树的差一错误。
最终结果是整体 IL 可能不同。希望更好。

•	当使用 /o+ 编译时，一些不必要的局部变量可能被消除或在栈上使用。
这不是新的，旧编译器也这样做，但此优化的实现已更改。
有些情况下旧编译器"更聪明"，因为它更早地进行此优化（并给后续阶段造成了许多不便）。
通常新方法处理更多场景，所以您可能会注意到在某些情况下使用了更多 dup 和更少的局部变量。

**级联优化** – 注意，一种优化可能启用另一种本身可能不是新的优化。
