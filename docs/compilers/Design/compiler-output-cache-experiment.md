# 编译器输出缓存实验

> [!WARNING]
> 此特性为**实验性**功能。它仅用于评估和试用，不是受支持的兼容性接口，可能存在 bug 或意外副作用，可能随时在不预先通知的情况下更改或移除。

## 概述

Roslyn 的编译器服务器已经通过在请求之间保持编译器状态存活来分摊部分编译开销。本实验将这一思路进一步延伸，允许服务器将成功编译的输出持久化到磁盘，并在后续请求中，当编译输入被确定为等效时恢复这些输出。

这对于在新的工作树中反复构建等效内容的工作流特别有用，包括代理驱动的开发场景——在这类场景中，即使 MSBuild 的常规最新检查跨克隆或临时工作目录不起作用，也可能存在匹配的编译。

当前实现有意被限定为一个实验：

- 仅对 `.NET 8+` 编译器服务器路径启用
- 主要为重复生成相同输出的本地或 CI 构建设计
- 预计随着我们对正确性、诊断和操作行为了解的深入而不断演进

## 目标

本实验有四个主要目标：

1. 当确定性输入未发生变化时，避免重复相同的编译。
2. 跨编译器服务器生命周期重用成功的编译器输出，而不是将重用限制在单个热进程内。
3. 通过在任何缓存未命中或缓存相关失败时回退到常规编译，来保证正确性。
4. 生成足够的日志，使缓存命中、未命中和键不匹配在实验评估期间可诊断。

## 非目标

当前设计**不**试图提供：

- 稳定的磁盘格式
- 跨 Roslyn 版本的长期兼容性承诺
- 带有已提交命令行开关或 SDK 行为的面向用户的产品特性
- 保证每个确定性构建都会从缓存中提供服务

## 启用方式

本实验仅编译到 `.NET 8+` 上的相关服务器端代码中。

在运行时，当编译器请求携带 `use-global-cache` 特性标志时，缓存才会启用。如果该特性标志不存在，服务器的行为与之前完全相同，不会缓存任何编译输出。

特性标志可以以下列任一形式使用：

- `use-global-cache`
- `use-global-cache=/absolute/path/to/cache`

当特性标志不带显式路径时，当前实现使用：

```text
<temp>/roslyn-cache
```

目前有两种方式将特性标志传递给请求：

1. 显式传递特性，例如通过 `/features:use-global-cache` 或 MSBuild 的 `Features` 属性。
2. 在调用编译器或构建之前，在客户端环境中设置 `ROSLYN_CACHE_PATH`。Roslyn 在构建编译器服务器请求时，将该值规范化为同一特性标志。

如果两者同时提供，显式特性标志优先于 `ROSLYN_CACHE_PATH`。

## 缓存键

缓存键基于 Roslyn 的确定性编译键。对于给定的编译请求，服务器从以下输入中计算确定性键：

- 编译内容
- 分析器
- 源代码生成器
- 附加文件
- 路径映射
- 输出选项
- 清单资源

该确定性键随后用 SHA-256 进行哈希，并用作缓存条目的目录键。

缓存还按输出文件名进行分组，因此有效的查找形式为：

```text
<cache root>/<output file name>/<sha-256-of-deterministic-key>/
```

## 缓存格式

每个缓存条目是一个目录，包含成功编译的输出以及用于生成这些输出的完整确定性键。

当前布局如下：

```text
<cache root>/
  <dll name>/
    <sha-256>/
      assembly
      pdb
      refassembly
      xmldoc
      <dll name>.key
      created
      last-used
```

其中：

- `assembly` 是主要输出的程序集，有效条目必须包含此项
- `pdb` 是输出的 PDB（如果存在）
- `refassembly` 是输出的引用程序集（如果存在）
- `xmldoc` 是输出的 XML 文档文件（如果存在）
- `<dll name>.key` 包含完整的确定性键文本
- `created` 包含首次存储时记录的 UTC 时间戳（`O` 格式，往返格式）
- `last-used` 包含每次缓存命中和初始存储时更新的 UTC 时间戳

只有在成功编译中实际存在的输出才会被存储。编译未生成的可选输出将被省略。

## 查找与填充流程

对于每个符合条件的编译请求，编译器服务器：

1. 计算确定性键。
2. 对该键进行哈希。
3. 在缓存根目录下查找现有条目。
4. 如果存在有效的缓存程序集，将缓存输出复制到请求的输出位置并报告成功。
5. 否则执行常规编译。
6. 成功编译后，尝试将输出发布到缓存以供未来请求使用。

读取或写入缓存时发生的任何故障都被视为缓存未命中或跳过存储，而非构建失败。请求回退到正常编译行为。

## 并发与发布

缓存条目通过暂存目录写入，然后通过目录移动进行发布。这旨在确保读取者只观察到完全填充的条目。

使用按条目同步，以确保在同一键上竞争的多个写入者不会同时尝试发布相同的结果。如果另一个进程已在填充某个条目，当前请求将直接继续正常编译行为。

## 诊断

在实验处于活跃状态期间，编译器服务器会记录重要的缓存事件，包括：

- 缓存已启用
- 缓存命中
- 缓存未命中
- 缓存存储成功
- 缓存存储被跳过（因为另一个写入者赢得了竞争）
- 缓存存储或恢复失败

当使用 `ROSLYN_CACHE_PATH` 时，客户端也会记录它将环境变量规范化为发送给服务器的特性标志。

在未命中时，Roslyn 还可能记录针对同一输出名称的最近先前键文件的差异。这旨在帮助解释为什么两个表面上相似的构建没有重用相同的缓存结果。

## 缓存管理

编译器服务器二进制文件（`VBCSCompiler`）提供用于检查和清理磁盘缓存的命令。这些命令供本地使用和 CI 集成使用，适用于实验评估期间。

### 命令

| 命令 | 描述 |
|------|------|
| `-cachestats` | 显示所有条目的缓存统计信息。 |
| `-cachestats:<timestamp>` | 显示缓存统计信息，将条目相对于给定 UTC 时间戳进行分类。 |
| `-cachestatsverbosity:<0\|1\|2>` | 控制 `-cachestats` 输出的详细程度。`0`（默认）只显示总计，`1` 按 DLL 名称分组，`2` 列出单个条目。 |
| `-purgecache` | 删除所有缓存条目。 |
| `-purgecache:<timestamp>` | 删除 `last-used` 时间早于给定 UTC 时间戳的缓存条目。 |
| `-cachepath:<path>` | 覆盖 `-cachestats` 和 `-purgecache` 的缓存目录（默认为标准缓存位置）。 |

时间戳使用 `DateTimeOffset.TryParse` 以 `InvariantCulture` 和往返种类进行解析，因此 ISO 8601 字符串（例如 `2026-04-22T10:00:00Z`）可以正常工作。

### `-cachestats` 如何对条目进行分类

`-cachestats` 上的可选时间戳是一个边界，用于将每个缓存条目分类到以下三个桶之一：

- **命中**——条目存在于时间戳*之前*，并在时间戳*之后*被*使用*（缓存重用）。
- **存储**——条目在时间戳*之后*被*创建*（导致新条目产生的缓存未命中）。
- **未触及**——条目存在于时间戳*之前*，且在时间戳*之后*未被使用。

未提供时间戳时，所有条目相对于时间起点（`DateTimeOffset.MinValue`）进行分类，因此每个条目都将显示为命中或存储。

### 工作流：了解一段时间内的缓存效率

要查看过去 30 天内缓存的效果：

```shell
VBCSCompiler -cachestats:2026-03-23T00:00:00Z
```

示例输出：

```text
Cache: /tmp/roslyn-cache
  Hits (reused):  450
  Stores (new):    50
  Untouched:      500
```

这告诉你 450 次编译是从缓存提供的，50 个新条目被存储，500 个旧条目在该时间段内未被使用。未触及数量正好是相应清除操作会删除的数量。

```shell
# 清除自该日期起未使用的条目
VBCSCompiler -purgecache:2026-03-23T00:00:00Z

# 删除所有内容，省略时间戳
VBCSCompiler -purgecache
```

### 工作流：CI 构建摘要

在 CI 中，在调用构建之前记录构建开始时间，然后用它获取该次特定构建期间缓存所做工作的摘要：

```shell
# 记录开始时间
BUILD_START=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

# 运行构建（启用缓存）
dotnet build -p:Features=use-global-cache

# 显示此次构建期间缓存的操作
VBCSCompiler -cachestats:$BUILD_START
```

由于时间戳设置为构建开始前的时刻，构建期间创建或使用的每个条目都被分类为该次构建的命中或存储。只要没有其他构建并行针对同一缓存运行，结果实际上就是每次构建的缓存摘要。

## 当前限制

本实验当前存在几个重要限制：

- 有效的缓存重用实际上需要 `PathMap` 稳定性。路径映射的差异会参与确定性键，因此不规范化机器特定源路径的构建在跨机器或工作树时命中同一缓存条目的可能性要低得多。
- 有效的重用还需要 `SourceLink` 稳定性。Source Link 输入的差异会参与确定性键，并会阻止跨构建匹配。这很可能意味着项目需要禁用 Source Link。
- 缓存命中时当前未完全保留正常的编译器行为。特别是，恢复的缓存结果当前不会重放在常规编译期间通常会产生的警告和其他编译器诊断信息。
- 由于缓存键基于当前的确定性键计算，控制缓存重用的输入集应被视为实验的实现细节，而非保证的约定。
- 磁盘缓存格式有意是临时的，可能在没有兼容性支持的情况下发生变化。

## 风险与待解问题

仍在评估的领域包括：

- 确定性键作为所有相关构建场景的缓存标识的正确性
- `PathMap` 和其他确定性输入的当前处理方式是否足以满足本实验预期有用的场景
- 磁盘增长问题，以及当前的手动清除命令是否应辅以自动清理
- 跨 SDK、编译器或机器环境变化的行为
- 当前请求级别的启用机制是否应长期保持当前形态
- 磁盘格式中哪些部分（如果有的话）应成为持久性的

由于这些问题仍处于开放状态，消费者应将本文档视为当前实验的描述，而非产品约定。
