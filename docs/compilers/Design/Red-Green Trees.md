# Roslyn 语法模型内部机制：红/绿树

本文档介绍 Roslyn 语法模型的内部架构，重点介绍驱动其高效性的"绿色"层。本文面向从事 Roslyn 编译器内部工作或与之打交道的开发者。

关于语法树的公共 API 和一般用法，请参阅官方文档：[使用 .NET Compiler Platform SDK 语法模型](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/work-with-syntax)。本文假设读者熟悉该材料，并深入探讨使语法模型高效运行的实现细节。

## 红/绿模式

Roslyn 的语法树使用一种有时被称为"红/绿树"的设计模式。其核心思想是维护同一语法结构的两种并行表示：

- **绿色节点**：内部不可变表示。这些节点只存储其语法*种类*、*宽度*（字符数）及其*子节点*。它们不存储绝对文本位置或父节点指针。

- **红色节点**：面向公众的包装器。它们提供带有 `Span`、`Parent` 和强类型子访问器等属性的熟悉 API。红色节点是薄包装器，将绿色节点与位置和父节点上下文结合起来。

这种分离的存在是因为不可变性和高效导航在根本上是矛盾的。不可变节点无法存储父节点指针（因为同一节点在不同上下文中可能有不同的父节点）。但用户需要父节点导航。红/绿模式解决了这个问题：绿色节点是不可变的、可共享的，而红色节点通过按需计算位置和父节点来提供导航 API。

## 绿色节点：核心设计

绿色节点存储：

- **种类**：这是什么类型的语法（例如，`MethodDeclaration`、`IdentifierToken`）
- **宽度**：此节点及其所有后代所跨越的字符总数
- **子节点**：对子绿色节点的引用（通过槽位）
- **诊断信息**：附加到此节点的任何语法错误

绿色节点**不**存储：

- **绝对位置**：绿色节点不知道它在源文件中的位置
- **父节点指针**：绿色节点不知道什么包含了它

这一设计选择是根本性的。所有其他一切都源于它。

### 基于槽位的子节点访问

绿色节点使用整数"槽位"来访问子节点。每种绿色节点类型都知道它有多少个槽位，以及每个槽位应该包含什么类型的子节点。例如，绿色 `MethodDeclarationSyntax` 节点可能有以下槽位：属性（0）、修饰符（1）、返回类型（2）、标识符（3）、类型参数（4）、参数（5）、约束（6）、方法体（7）等。

对于每种红色语法节点类型，都有一个对应的内部绿色语法节点类型。因此既有公共红色 `MethodDeclarationSyntax`，也有内部绿色 `MethodDeclarationSyntax`。绿色类型可以通过槽位索引访问（用于统一遍历），也可以通过与每个子节点对应的强类型属性访问。红色层在通过红色 API 生成子节点时直接访问这些绿色属性。

这种设计使内部算法能够通过槽位索引泛型地遍历绿色树（无需了解正在访问的节点类型），同时在节点类型已知时提供高效的强类型访问。

## 无位置、无父节点节点的优势

省略绿色节点中的位置和父节点的决定启用了几种强大的优化。

### 增量解析与子树重用

当用户编辑文件时，增量解析可以重用之前语法树的大部分内容。由于绿色节点不编码绝对位置，位于位置 10,000 的方法声明在较早编辑之后可以在位置 10,050 处被重用。绿色节点是相同的；只有红色包装器的计算位置发生变化。

这使得增量解析能够在微秒内完成，对于典型编辑，内存重用率接近 99.99%。详细信息请参阅 [Roslyn 增量解析：深度解析](./Incremental%20Parser.md)。

### 跨树共享

同一绿色节点可以出现在完全不相关的语法树中。考虑空参数列表 `()` 的 `ParameterList` 节点。这个确切的结构出现在大型解决方案的数千个方法中。由于绿色节点没有位置或父节点，*同一个*绿色节点对象可以在所有这些地方共享。

这种共享通过绿色节点缓存自动发生（如下所述）。结果是在解决方案级别实现了显著的内存节省。

### 树内共享：它是有向无环图，而非树

同一绿色节点可以在*单个*语法树中出现多次。如果一个文件包含十个方法，它们都有空参数列表 `()`，那么十个方法都可以指向同一个 `ParameterList` 绿色节点。

这意味着"绿色树"实际上是一个误称。绿色结构是一个**有向无环图**（DAG），而非树。多个父节点可以共享同一个子节点。红色层隐藏了这种复杂性。每个红色节点看起来都是具有自己唯一父节点和位置的独立对象，即使底层绿色节点可能是共享的。

## 绿色层的所有内容都是节点

在公共红色 API 中，语法模型对不同概念使用不同类型：

- `SyntaxNode`：语法节点的类（引用类型）
- `SyntaxToken`：token 的结构体（值类型）
- `SyntaxTrivia`：空白符和注释的结构体
- `SyntaxList<T>`：节点列表的结构体

在绿色层中，**所有这些都是堆分配的节点对象**。没有结构体。每个绿色 token、每个绿色 trivia、每个绿色列表都是 `GreenNode` 的子类。

为什么？因为结构体无法参与上述共享场景。如果 token 在绿色层是结构体，在多个位置重用一个标识符就需要每次复制该结构体。使用绿色节点作为对象，我们可以指向同一个实例。

除了共享之外，这一选择还启用了绿色层设计的一个重要方面：特化。绿色树包含许多针对不同场景量身定制的特化实现（例如，优化的 token 表示和特化的列表实现）。这种特化对于实现良好的性能和内存使用至关重要，下文将详细介绍。

结构体不适合这一角色，因为它们无法参与继承层次结构，也无法在公共抽象后支持多种特化实现。通过将绿色节点表示为对象，我们可以使用标准继承技术自由地特化内部表示，同时仍在公共语法层提供简单、轻量级的基于结构体的 API。公共结构体有效地充当这些底层绿色节点的薄外观。

结果是红色包装器极为廉价。红色 `SyntaxToken` 结构体只包含：

- 指向父红色节点的指针
- 一个整数位置
- 指向底层绿色 token 节点的指针

访问 token 时不会发生堆分配。同样适用于红色层的 trivia 和列表。

由于 token、列表和 trivia 通常占语法树元素的 **75% 或更多**，这种设计避免了遍历树时原本会发生的绝大多数分配。

## 列表优化

列表在语法树中无处不在：参数列表、参数列表、语句列表、修饰符列表等。Roslyn 在绿色层对列表表示进行了大量优化。

### 空列表

空列表在绿色层表示为 `null`。完全没有分配。当被红色 `SyntaxList<T>` 包装时，它检查 null 并返回 `Count = 0`。

### 单元素列表

恰好有一个元素的列表不分配绿色列表节点。相反，父绿色节点直接指向单个子元素。红色 `SyntaxList<T>` 检测到它指向的是非列表绿色节点，返回 `Count = 1`，只提供对该元素的访问。

### 小列表（2、3、4 个元素）

长度为 2、3 和 4 的列表有特化的绿色节点子类（`WithTwoChildren`、`WithThreeChildren` 等）。这些子类将子节点指针直接存储在字段中，而不是使用数组。这避免了数组分配和访问数组元素的间接层。

### 较大的列表

包含 5 个或更多元素的列表使用基于数组的表示。

#### 为什么这很重要

实际上，**90% 或更多的列表包含 4 个或更少的元素**。想想你写带有 0、1、2 或 3 个参数的方法与带有 5 个以上参数的方法的频率。参数列表、类型参数列表、属性列表——所有这些都遵循类似的分布。

通过针对常见情况进行优化，Roslyn 避免了绝大多数列表的数组分配。红色 `SyntaxList<T>` 和 `SeparatedSyntaxList<T>` 结构体透明地处理所有这些情况，无论底层表示如何，都提供统一的 API。

### 弱引用持有的红色子节点

某些红色列表能够使用弱引用持有其红色子节点。这允许当红色树的某些部分不再被强引用时，由 GC 回收它们，而不影响正确性。

这是安全的，因为红色节点只有在没有任何东西持有对它的强引用时才能被回收。如果唯一剩余的引用是父列表持有的弱引用，则没有任何可能的观察者能够注意到红色节点的消失。如果之后再次请求该子节点，将从同一底层绿色节点创建一个新的红色节点。由于红色节点纯粹是不可变绿色结构的缓存，这种行为是不可观察的，在语义上等同于保留原始红色节点。

#### 当前用法

目前，此优化仅应用于由成员或访问器拥有的块（`{ ... }`）（[参见此处](https://github.com/dotnet/roslyn/blob/744249e4f0afb645808e87aebec109a38d5dde8b/src/Compilers/CSharp/Portable/Syntax/CSharpSyntaxNode.cs#L518)）。这些块可能很大，通常只在分析期间临时需要。通过允许此类块的红色列表弱引用其子节点，可以在没有其他内容持有这些节点时释放大量红色内存，从而降低峰值内存使用，同时保留现有的红/绿树语义。

## 绿色节点缓存

Roslyn 维护一个常见绿色节点的缓存。在解析过程中构建新绿色节点时，解析器首先检查缓存中是否已存在等效节点。如果存在，则重用缓存的节点而不是分配新的。

由于组合爆炸，只有**3 个或更少子节点**的节点才有资格缓存。缓存本身包含 65,536 个条目。尽管有这些限制，对解析 Roslyn 代码库本身的分析显示，可缓存节点的**缓存命中率为 55%**。这意味着超过一半的时间，解析器可以重用现有的分配，而不是创建新的。

有关实现细节，请参阅 [PR #80825](https://github.com/dotnet/roslyn/pull/80825)，该 PR 简化并分析了 `SyntaxNodeCache`。

这种缓存正是实现前述跨树和树内共享的原因。`()` 参数列表不仅在理论上可共享——它*实际上*是共享的，因为每次解析该结构时，缓存都返回同一个节点。

## Token 存储优化

由于绿色节点是内部的，我们可以自由优化其存储，而这些优化公开暴露会很尴尬。Token 很好地说明了这一点。

### 标识符 Token

标识符 token 存储指向其文本（例如 `"myVariable"`）的指针。由于存储了文本，宽度不需要单独存储——可以从字符串长度计算出来。这为树中每个标识符节省了空间。

### 关键字 Token

关键字 token 的工作方式相反。所有 `return` 关键字都有完全相同的文本，因此不需要存储它。相反，关键字 token 只存储其*种类*，文本和宽度从中推断出来。一个 `return` 关键字 token 知道它是 `SyntaxKind.ReturnKeyword`，文本 `"return"` 和宽度 `6` 自动得出。

### 预缓存的关键字变体

Roslyn 预先计算并缓存每个不带 trivia 的关键字 token，以及带有常见前导和尾随空白模式的变体。考虑 `void ` （关键字 `void` 后跟尾随空格）。这个确切的结构在源文件中频繁出现，每次出现都指向同一个字面绿色节点实例。解析器无需分配任何内容——它只是返回预缓存的节点。

这些优化对红色 API 的用户不可见，但它们在解析期间显著减少了内存使用和分配压力。

### 预缓存的 Trivia

同样的缓存策略适用于 trivia。单个空格、换行符或典型缩进序列（两个空格、四个空格、八个空格、制表符序列等）等常见模式在文件内和跨文件中出现无数次。Roslyn 预缓存这些常见模式，而不是每次分配新的绿色 trivia 节点。解决方案中每个单空格 trivia 的出现都指向同一个绿色节点实例。

## 红色节点创建：惰性且缓存

红色节点在你遍历语法树时按需创建。如果你从未访问某个特定节点，其红色包装器就永远不会被创建。

对于红色 `SyntaxNode` 实例（类类型），创建后需要缓存。这是必要的，因为节点具有引用标识——用户期望两次访问同一子节点返回同一对象。缓存很简单：当红色父节点被要求提供子节点时，它检查子红色节点是否已经创建。如果没有，它实例化子节点并原子性地将其写入字段存储。赢得竞争的任何线程都会设置所有后续读取将看到的指针。

对于红色 `SyntaxToken`、`SyntaxTrivia` 和 `SyntaxList<T>`（所有结构体类型），不需要缓存。每次访问返回一个新的结构体实例，但结构体具有值语义，所以这没问题。指向同一绿色节点在同一位置的两个红色结构体被认为是相等的。

### 对性能的影响

不遍历整个树的特性只需为它们实际触及的部分付出代价。考虑 IDE 代码着色：它只需要检查屏幕上可见的 token。滚动离屏的方法的红色节点永远不会被创建（或者，如果之前已创建，可能在内存压力下被回收）。

结合增量解析的大量绿色重用，这意味着语法层在正常使用模式下分配很少：

- 绿色节点在编辑间和文件间被大量重用
- Token、列表和 trivia（占树的 75% 以上）没有红色分配
- 由于惰性创建，红色节点分配最小化
- 只有被主动检查的树的那部分才会产生红色节点代价

## 不可变性

绿色节点是完全不可变的。一旦创建，绿色节点就永远不会改变。这有几个好处：

- **线程安全性**：多个线程可以在无锁的情况下读取同一语法树
- **可预测性**：一旦你有了对节点的引用，它就不会在你的眼皮底下改变
- **共享性**：不可变对象可以自由共享，无需防御性复制

当你需要"修改"语法树（通过 `WithXxx` 方法或 `SyntaxFactory`）时，你得到一棵新树，它尽可能多地与旧树共享结构。只有从根到修改点路径上的节点才是新分配的；其他所有内容都被重用。

有关使用不可变语法树的更多信息，请参阅[官方文档](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/work-with-syntax)。

## 完全保真

Roslyn 语法树是*完全保真*的：源文件中的每个字符都在树的某处有所体现。空白符、注释、预处理器指令，甚至格式错误的语法都被保留。按顺序拼接所有 token 和 trivia，可以精确地重现原始源代码。

这一属性对于工具场景至关重要。重构工具可以修改树的特定部分，同时保留用户在其他地方的格式。错误恢复确保部分编写的代码仍然产生可导航的树。

完全保真还使增量解析能够精确推理树的哪些部分受编辑影响。详细信息请参阅 [Roslyn 增量解析：深度解析](./Incremental%20Parser.md)。

### 设计哲学

代码库包含许多针对不同节点配置、列表大小和 token 类型的特化子类。这允许在绿色层进行大量优化。然而，红/绿分离的力量在于，所有这些复杂性对用户来说都是隐藏的。公共 API 呈现一棵简洁、直观的语法树，带有父节点导航和绝对位置。用户不需要了解绿色节点、基于槽位的访问、列表表示技巧或缓存策略。

然而，由于这些内部优化，结合上述使用模式，性能是出色的：

- Token、列表和 trivia 不产生红色分配
- 由于惰性创建，红色节点分配最小化
- 绿色节点在树内和跨树大量共享
- 增量解析跨编辑几乎重用了所有内容

复杂性存在于实现中，这样用户就能获得令人愉悦的 API 和出色的性能。
