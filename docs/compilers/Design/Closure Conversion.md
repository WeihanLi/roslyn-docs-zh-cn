C# 中的闭包转换
========================

本文档描述了 C# 编译器如何将闭包（匿名函数和局部函数）转换为顶级函数（方法）。

如果您不熟悉闭包转换，介绍部分包含一个描述转换的演练。否则，您可以跳到内部部分查看 Roslyn 中实际如何执行转换。

# 介绍

在最简单的情况下，这是微不足道的——所有闭包只是被赋予新的、不可提及的名称，并作为静态方法提升到顶级。当闭包从其周围作用域捕获变量时，复杂性就来了。在那一点上，我们不仅必须将闭包移动到方法，还必须创建一个"环境"来保存其捕获的变量，并以某种方式将该环境传递到重写方法的上下文中。

编译器在不同情况下采用两种可能的策略。第一种策略是将环境作为额外参数传递给所有对闭包的方法调用。例如，

```csharp
void M()
{
    int x = 0;
    int Local() => x + 1;
}
```

变成

```csharp
void M()
{
    var env = new Environment();
    env.x = 0;
    <>__Local(env);
}

static int <>__Local(Environment env)
{
    return env.x + 1;
}

struct Environment
{
    int x;
}
```

重写的闭包现在不是引用局部变量，而是引用环境上的字段。使用 `struct` 环境来防止额外分配，因为 `struct` 通常在栈上分配。然而，精明的读者可能会注意到这种转换的一个问题——`struct` 在作为函数调用的参数传递时总是被复制！这意味着对环境字段的写入不会总是传播回方法 `M` 中的局部变量。为了解决这个缺陷，所有环境变量都作为 `ref` 参数传递给重写的闭包。

重写闭包的第二种策略在闭包与外部代码交互时是必要的。考虑以下程序：

```csharp
void M(IEnumerable<int> e)
{
    int x = 0;
    var positive = e.Where(n => n > 0);
    ...
}
```

在这种情况下，我们将闭包传递给 `IEnumerable<int>.Where` 函数，它期望一个 `Func<int, bool>` 类型的委托。注意，该委托类型是不可变的——它在外部代码中定义，不能更改。因此，重写外部调用点以接受额外的 `Environment` 参数是不可能的。我们必须选择不同的策略来获取环境。

我们对这种情况使用 `class` 环境。使用 `class` 环境，前面的环境可以重写如下：

```csharp
void M(IEnumerable<int> e)
{
    var env = new Environment();
    env.x = 0;
    var positive = e.Where(env.<>__Local);
}

class Environment
{
    int x;

    bool <>__Local(int n) => n > 0;
}
```

由于局部变量现在是类实例中的字段，我们可以保持相同的委托签名，并依赖字段访问来读写自由变量。

这涵盖了 C# 在高级别执行的转换。以下部分详细介绍了如何执行这些转换。

# 内部

闭包转换的顶级有两个阶段。第一个阶段，分析，负责构建一个 AnalysisResult 数据结构，其中包含重写所有闭包所需的所有信息。第二个阶段，重写，实际上通过替换和添加 BoundNodes 到绑定树来执行前面提到的重写。这两个阶段之间最重要的契约是分析阶段不对绑定树进行任何修改，重写阶段不执行任何计算，除了基于 AnalysisResult 对绑定树进行简单的机械修改外不包含任何逻辑。

## 分析

在此阶段，我们构建一个 AnalysisResult，它是一个精确表示从原始程序到闭包转换形式的状态映射的树结构。

例如，以下程序

```csharp
void M()
{
    int x = 0;
    int Local() => x + 1;
    {
        int y = 0;
        int z = 0;
        int Local2() => Local() + y;
        z++;
        Local2();
    }
    Local();
}
```

将生成类似于

```
  +-------------------------------------+
  |Captured:       Closures:            |
  |int x           int Local()          |
  |int Local()                          |
  |                                     |
  | +--------------------------------+  |
  | | Captured:      Closures:       |  |
  | | int y          int Local2()    |  |
  | |                                |  |
  | +--------------------------------+  |
  +-------------------------------------+

```

的树

要创建此 AnalysisResult，需要多个遍历。第一个遍历通过构建作用域、闭包和捕获变量的朴素树来收集信息。结果是嵌套作用域的树，其中每个作用域列出在该作用域中声明的捕获变量和该作用域中的闭包。第一个遍历必须首先收集信息，因为大多数重写决策，如对闭包使用什么环境类型或重写的闭包签名是什么，都依赖于整个方法的上下文。

关于捕获变量的信息存储在 `CapturedVariable` 类的实例上，它保存有关被捕获的符号的信息和重写信息，如重写后将替换变量的 `SynthesizedLocal`。类似地，闭包将存储在 `Closure` 类的实例中，其中包含原始符号和为闭包创建的合成类型或方法。所有这些类都是可变的，因为预期后续遍历将使用从早期遍历收集的结构填充更多重写信息。

例如，在前面的示例中，我们需要遍历树来为闭包生成环境 `Closure`，结果类似于以下内容：

```
Closure
-------
Name: Local
Generated Sig: int <>_Local(ref <>_Env1)
Environment:
    - Captures: 'int x'
    - Name: <>_Env1
    - Type: Struct

Name: Local2
Generated Sig: int <>_Local(ref <>_Env2, ref <>_Env1)
Environment:
    - Captures: 'int y', 'ref <>_Env1'
    - Name: <>_Env2
    - Type: Struct
```

此结果将由每个部分填充所需信息生成：首先填充所有捕获列表，然后根据捕获列表决定环境类型，然后根据环境类型和最终捕获列表生成最终签名。

下面可以找到分析计算的一些更多详细信息：

**待办** _随着实现的完善添加每个分析阶段的详细信息_


* 决定每个闭包需要什么环境类型。环境可以是 struct，除非以下情况之一为真：

    1. 闭包被转换为委托。
    2. 闭包捕获一个被不能有 `struct` 环境的闭包捕获的变量。
    3. 对闭包的引用被不能有 `struct` 环境的闭包捕获。

* 为提升的捕获变量和捕获的环境引用创建 `SynthesizedLocal`
* 为每个闭包分配重写的名称、签名和类型参数

## 优化

最后的遍历是优化遍历。它们试图通过删除没有捕获变量的中间作用域和以其他方式更正树来简化树，以创建最有效的重写形式。优化机会可能包括运行逃逸分析以确定捕获提升是否可以通过复制完成或在更窄的作用域中完成。

## 重写

重写阶段只是遍历绑定树，在每个新作用域，检查 AnalysisResult 是否包含要处理的重写信息的作用域。如果是，局部变量和闭包将被树提供的替换所替换。实际上，这意味着树包含：

1. 要添加到每个作用域的合成局部变量列表。

2. 要替换现有符号的代理集合。

3. 要添加到封闭类型的合成方法和类型列表。
