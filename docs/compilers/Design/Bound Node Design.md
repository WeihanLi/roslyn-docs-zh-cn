绑定节点设计
=================

本文档讨论绑定节点的设计原则。

### 绑定树的形状应对应于程序静态语义的形状

一般来说，这意味着语法和绑定节点之间存在同构关系，除非语法和语义的形状不匹配，在这种情况下它们建模语义的形状。在可能的情况下，我们更喜欢对应尽可能直接。以下是两个说明这一点的示例：
1. 括号表达式不出现在绑定节点中，因为它们没有语义意义。
2. 查询表达式通过对应于翻译形式被赋予语义意义，因此绑定节点可以建模翻译形式。

绑定节点的默认访问顺序应匹配求值顺序，这通常匹配词法顺序。

### 绑定节点应捕获嵌入在语法中的所有语义信息

绑定节点的使用者不应该需要检查产生它们的语法来理解绑定节点的含义。来自语法的所有相关语义信息应该在绑定节点的字段中总结。如果绑定节点的使用者需要引用语法来影响代码的含义，那是一种设计异味。

### 每个绑定节点类型要么是抽象的，要么是密封的

在 `BoundNodes.xml` 中，`Node` 应该有一个基类是 `BoundNode` 或声明为 `AbstractNode` 的节点类型。不要从另一个具体节点类型继承具体节点类型。
