解析器设计指南
========================

本文档描述了 Roslyn 解析器的设计指南。这些不是不能违反的硬性规则。请使用良好的判断力。当有理由超过其好处时，可以与指南有所不同。

![image](https://i0.wp.com/media.tumblr.com/tumblr_lzwm2hKMGx1qhkwbs.gif)

解析器目前不符合这些指南，即使在没有充分理由不遵守的地方也是如此。这些指南是在解析器之后编写的。我们期望在有具体好处时有机会地重构解析器以符合这些指南。

为新功能设计语法模型（语法树的数据模型）目前不在这些指南的范围内。

#### **要** 让解析器接受符合语法模型的输入文本

语法模型定义了语法和语义分析之间的契约。如果解析器可以为一系列输入标记构建有意义的语法树，那么应该将报告该树在语义上无效的工作留给语义分析。

解析器报告的最重要的诊断是"缺少标记"和"意外标记"。

可能有理由违反此规则。例如，语法模型没有优先级的概念，但解析器使用优先级来决定如何组装树。因此，优先级错误在解析器中报告。由于树的语法形状在这里受到影响，我们不想形成非法树，因此在此层的诊断是合理的。

但是，对于可以轻松适应语法模型但规则实际上是由外部上下文驱动的语义规则的情况，应避免诊断。一个很好的例子是"语言版本"检查。除了特殊情况（如"记录解析"）之外，解析实际上不受语言版本的影响。相反，语言版本只是说明该构造是否可以使用，而不是它是否被识别并成功解析到语法模型中。这些检查应该在稍后进行，比如在编译器的声明表或绑定过程中（请参阅本文档下面的 DO/DON'T 部分了解原因）。注意：在解析器级别仍然保留这些检查的可接受原因是避免简单直接的解析检查变成更高级别的"涂抹花生酱"检查（例如，可能需要在几十个位置添加检查）。在这种情况下，对编译器可维护性的负面成本将超过我们在其他地方获得的正面好处。

#### **不要** 使用环境上下文来指导解析器的行为

解析器应尽可能地与上下文无关。仅在需要解决语言歧义时使用上下文。例如，在类型和表达式都允许但首选类型的上下文中（如 `is` 的右侧），我们将 `X.Y` 解析为类型。但是在两者都允许但只有在后跟标识符时才视为类型的上下文中（如 `case` 之后），我们使用前瞻来决定要构建哪种类型的树。尽量减少用于做出解析决策的上下文量，因为这将提高增量解析的质量。

可能有理由违反此规则，例如当语言被指定为对上下文敏感时。例如，如果封闭方法具有 `async` 修饰符，`await` 被视为关键字。

#### 不遵循这些 DO/DON'T 的影响

上述 DO/DON'T 的原因与语法或解析器设计的感知纯度无关。相反，它们对编译器的可维护性和正确性以及直接使用语法模型的堆栈更高部分的性能有重大影响（IDE 是这里的主要使用者）。具体来说：

1. 环境上下文的存在极大地影响了正确进行增量解析的能力。必须以某种方式跟踪环境上下文，跨不同上下文的增量重用会导致违反增量重新解析产生与正常解析相同的树的不变量。这可能会导致只能通过主机重启解决的根本性故障。例如，考虑一个向方法添加 'async' 的编辑。这必然会影响方法内节点的重用，因为它们可能被单独解析。每一条环境上下文都可能产生这种效果，这使得推理增量编辑变得更加困难，并可能导致在野外难以诊断或重现的微妙增量错误。

2. 超出"缺少标记"和"意外标记"的诊断的存在再次影响增量解析器。增量解析无法重用包含错误的节点，因为它不知道错误的根本原因是什么，以及文件不同部分的编辑是否会修复它。因此，任何额外的诊断都会强制完全重新解析该构造及其上方的所有父节点。这会增加增量解析的额外时间，并可能导致更高的内存流失，因为必须创建新节点。这两个问题都直接影响 IDE 场景中的下游使用。在 IDE 中，打字延迟是我们需要满足的高价值 SLA 目标。但是，许多功能必须在打字窗口内以与语法树"正确"的方式运行。这包括但不限于括号插入和缩进等功能。这两者都必须对用户近乎即时地发生。而且两者都需要一个最新的 SyntaxTree 来正确确定该做什么。因此，这些树是使用增量解析同步检索的，因此必须尽可能少地进行额外工作，以便我们可以将所有剩余工作放入可用的时间片中。额外的诊断会干扰这一点，增加额外不必要的 CPU 并导致内存流失。

3. 许多 IDE 功能采取的观点是它们不能弊大于利。核心方法是假设如果树格式不正确，那么从语法上调整它可能会导致问题。这影响了不想使糟糕情况变得更糟的自动格式化等功能。通过尽可能将诊断限制为"缺少标记"和"意外标记"，这些更高级别的功能更容易提出问题："树是否以应该阻止我继续的方式损坏？"但是，如果树符合语法模型但包含与语法形状可行性无关的*其他*诊断（如"这是否在错误的语言版本中使用"的语义问题），那么更高级别的功能会被不必要地禁用。更新这些功能以使用编译器错误代码是脆弱的，并且有效地将它们与可以简单查询的 API 联系起来，而不是关于诊断的不透明信息（如它们位于何处以及为什么发出它们）。这类似于增量解析面临的问题，只是针对更高级别的组件。

#### 示例

以下是一些我们可能会更改解析器以满足这些指南的地方的示例，以及可能解决的问题：

1. 100l：警告：对长字面量使用 `L`

   在解析器中产生此警告似乎是一件小事，但它会干扰增量解析。增量解析器不会重用具有诊断的节点。因此，即使代码中存在此有用的警告也可能会降低用户在该源代码中键入时解析器的性能。性能降低可能意味着编辑器看起来不那么响应。

   通过将此警告从解析器移到语义分析中，IDE 可以在键入期间更加响应。

1. 成员解析

   语法模型使用单独的节点表示构造函数和方法。当看到 `public M() {}` 形式的声明时，解析器会将函数名称与封闭类型的名称进行比较，以决定是否应将其表示为构造函数，还是作为缺少类型的方法。在这种情况下，封闭类型的名称是影响编译器行为的*环境上下文*的一种形式。

   通过在语法模型中将此类声明视为构造函数，并在语义分析中检查名称，就可以公开用于解析成员声明的与上下文无关的公共 API。请参阅 https://github.com/dotnet/roslyn/issues/367 了解由于此问题我们无法完成的功能请求。
