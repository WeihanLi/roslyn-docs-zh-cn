# Roslyn 分析器获取策略

本文档描述了使用者应如何随时间推移获取 **Microsoft.CodeAnalysis.*（Roslyn）分析器包**，以及获取模型在各主要版本之间的变化原因。

## 目标

1. **可靠的分析器执行**：分析器应在与其兼容的编译器/宿主上运行。
2. **降低维护成本**：减少长期支持极旧编译器宿主所带来的负担。
3. **简化获取方式**：从"添加 NuGet 包"逐步过渡到"通过 MSBuild 属性启用"。
4. **版本对齐**：分析器应与执行它们的**编译器版本匹配**。

---

## 推荐指导

### 如果需要最大宿主兼容性
- 优先选用从 `dotnet/roslyn-analyzer` 构建的 **<= 3.11.0** 版本分析器包，尤其是当构建必须在旧编译器/宿主上运行时。

### 如果使用现代 SDK 并希望获取最新分析器版本
- 使用 **5.0+** 版本分析器包，并确保构建使用 **.NET 9 SDK 或更高版本**。

### 为基于 SDK 的未来做好规划
- 预期从基于 `PackageReference` 的获取方式向 **SDK 内置分析器**（通过配置属性启用）过渡。
- 优先使用集中配置（例如 `Directory.Build.props`）以便未来迁移。

---

## 背景与时间线

### <= 3.11.0 版本的分析器包

对于**小于或等于 3.11.0** 的包版本：

- 分析器包构建自 **`dotnet/roslyn-analyzer`** 仓库。
- 这些版本旨在支持在**非常旧的 Roslyn 编译器和编译器宿主**（例如旧版 Visual Studio / 旧版 SDK 工具集）上运行。
- 这需要大量的**向后兼容性代码和测试**，增加了维护开销。

**对使用者的影响：** 这些包与旧工具链广泛兼容，是分析器执行必须在旧环境中正常工作时的理想选择。

---

### 5.0+ 版本的分析器包（合并进 dotnet/roslyn 之后）

**`dotnet/roslyn-analyzer`** 仓库已合并到 **`dotnet/roslyn`**。作为整合的一部分：

- 大量针对极旧宿主的特殊兼容性逻辑已被**移除**，以降低维护负担。
- 新版本包采用与现代 SDK/编译器发布对齐的兼容性策略。

对于新的 **5.0** 版本包：

- 它们**要求 Visual Studio 17.12 / .NET 9 SDK 或更高版本中的编译器**。

**对使用者的影响：** 使用 5.0 版本分析器意味着你的构建使用了足够新的编译器/SDK。如果构建环境较旧，这些分析器不是受支持的选择。

---

## 当前获取模型（现状）

目前，获取方式通常通过 **NuGet PackageReference** 完成，例如：

- 将分析器包添加到项目（`.csproj`）或集中在 `Directory.Build.props` 中。
- 确保构建环境使用足够新的编译器/SDK 来运行该分析器版本（尤其是 5.0+ 版本）。

这种模式有效，但存在以下缺点：

- 使用者需要自行发现并选择包版本。
- 可能发生版本偏差：分析器可能比执行它们的编译器更新或更旧。
- 升级分析器可能隐式要求升级 SDK/编译器宿主。

---

## 未来方向：在 .NET SDK 中内置分析器

长期目标是将这些分析器包**内置于 .NET SDK 中发布**。目前，Roslyn 已在 SDK 中发布其 CodeStyle 分析器，使用者通过 `EnforceCodeStyleInBuild` 属性选择启用。

### 为何这更好

当分析器随 SDK 一起发布时：

- 分析器可以**自动与编译器版本匹配**。
- 获取体验变得更简单：
  - 无需添加 `PackageReference`，而是通过在 `.csproj` 或 `Directory.Build.props` 中设置 **MSBuild 属性**来启用分析器。
- SDK 可以确保分析器/编译器配对经过验证，减少兼容性意外。

### 对使用者意味着什么

从概念上讲，使用者将从：

- "通过 NuGet 添加分析器包"

转变为：

- "通过属性启用分析器"

并在以下位置进行配置：

- 项目文件（`.csproj`）
- `Directory.Build.props`（推荐用于仓库级策略）

*（具体属性名称和最终用户体验可能随 SDK 集成方案的完善而调整。）*
