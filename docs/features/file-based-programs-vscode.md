# 基于文件的程序 VS Code 支持

另请参见 [dotnet-run-file.md](https://github.com/dotnet/sdk/blob/main/documentation/general/dotnet-run-file.md)。

## 功能概述

基于文件的程序将 MSBuild 项目功能的子集嵌入到 C# 代码中，允许单个文件作为普通项目独立存在。

以下是一个基于文件的程序：

```cs
Console.WriteLine("Hello World!");
```

以下也是：

```cs
#!/usr/bin/env dotnet run
#:sdk Microsoft.Net.Sdk
#:package Newtonsoft.Json@13.0.3
#:property LangVersion=preview

using Newtonsoft.Json;

Main();

void Main()
{
    if (args is not [_, var jsonPath, ..])
    {
        Console.Error.WriteLine("Usage: app <json-file>");
        return;
    }

    var json = File.ReadAllText(jsonPath);
    var data = JsonConvert.DeserializeObject<Data>(json);
    // ...
}

record Data(string field1, int field2);
```

这基本上是通过让 `dotnet` 命令行解释源文件中的 `#:` 指令、在内存中生成 C# 项目 XML 文档，然后将其传递给 MSBuild 来实现的。内存中的项目有时被称为"虚拟项目"。

## 丰富的杂项文件

长期以来，有一个积压项目是增强处理杂项文件（"松散文件"，不与任何项目关联）的体验。我们认为，作为"基于文件的程序"工作的一部分，我们可以在此类文件中无实质性问题地启用以下功能：
- 语法诊断。
- "默认"引用集的 Intellisense，例如，`dotnet new console` 使用当前 SDK 创建的项目中包含的引用。
- 在某些情况下，我们甚至可以启用语义诊断，并有合理的置信度认为产生的错误对用户有用。

对杂项文件行为的这些更改称为"丰富杂项文件"。

实现策略是：编辑器在临时目录下创建一个"规范杂项文件项目"，并使用生成的项目信息作为在 IDE 中打开的松散文件的"基础项目"。

## 基于文件的应用检测

C# 文件在编辑器中有多种可能的*分类*：
- **基于项目的应用**。文件是普通 `.csproj` 项目的一部分。
- **基于文件的应用**。文件是"基于文件的应用"项目的一部分，即它是基于文件的应用的入口点，或者被同一应用的入口点通过 `#:include` 包含。
- **带标准引用和语义错误的杂项文件**。文件是基于文件的应用的有效入口点，但缺少让我们高度确信这是用户意图的 `#:`/`#!` 指令。
   - 工具将相应地启用，显示语法错误、语义错误、核心库的语义信息等。请参见上面的*丰富杂项文件*部分。
   - 这些文件不会被还原。
- **带标准引用的杂项文件**。文件不属于任何项目，且启发式判断表明它不打算作为基于文件的应用。该文件使用常规 C# 语言（而非 `.csx` 脚本方言）。
   - 语法错误和核心库的语义信息将在这些文件中显示。
   - 语义错误不会在这些文件中显示。
- **无引用的杂项文件**。文件不属于任何项目。它甚至可能是 `.csx`、`.razor` 或其他非 .cs 类型。
   - 这些文件没有对核心库的引用，也不显示语义错误。
   - 语法错误、跳转到同一文件中声明的定义等可能有效。
   - 当 `enableFileBasedPrograms` 被禁用时，通常使用此分类，而不是上面的*带标准引用的杂项文件*或*基于文件的应用*分类。

**注意：** 这旨在作为一个动态文档，检查集和分类可能会根据我们的需求随时间变化。

以下是确定如何对 C# 文件进行分类的决策树：

1. **文件是否在当前加载的项目中？**
   - **是** → 分类为**基于项目的应用**
   - **否** → 继续下一项检查

2. **是否启用了 `enableFileBasedPrograms`？**（默认：在发布版中为 `true`）
   - **否** → 分类为**无引用的杂项文件**
   - **是** → 继续下一项检查

3. **文件是否是常规 C# 文件？（即不是 `.csx` 脚本，也不是使用 C# 以外语言的文件）**
   - **否** → 分类为**无引用的杂项文件**
   - **是** → 继续下一项检查

4. **文件是否具有绝对路径且存在于磁盘上？**（即它不是为新的、尚未保存的文件或类似情况创建的"虚拟文档"。）
   - **否** → 分类为**带标准引用的杂项文件**
   - **是** → 继续下一项检查

5. **文件是否有 `#!` 指令？**
   - **是** → 分类为**基于文件的应用**。根据需要还原并显示语义错误。
   - **否** → 继续下一项检查

6. **文件是否有 `#:` 指令？**
   - **否** → 转到 (8)
   - **是** → 继续下一项检查

7. **文件是否有顶级语句？**
   - **是** → 分类为**基于文件的应用**。根据需要还原并显示语义错误。
   - **否** → 分类为**带标准引用的杂项文件**

8. **是否启用了 `enableFileBasedProgramsWhenAmbiguous`？**（默认：在发布版中为 `false`，在预发布版中为 `true`）
   - **否** → 分类为**带标准引用的杂项文件**
   - **是** → 继续启发式检测

**启发式检测（当 `enableFileBasedProgramsWhenAmbiguous: true` 时）：**

9. **文件是否有顶级语句？**
   - **否** → 分类为**带标准引用的杂项文件**
   - **是** → 继续下一项检查

10. **文件是否包含在 `.csproj` 范围内？**
   - "范围"意味着某个包含目录（在某个嵌套级别）中有一个 `.csproj` 文件。
   - 注意，此特定检查仅在文件打开时执行。我们认为典型情况是用户将加载他们正在创建的新项目。加载项目将导致文件按 (1) 开始被视为基于项目的应用。如果用户不加载新项目，则在文件关闭并重新打开之前，过时的诊断可能仍然存在。
   - **是** → 分类为**带标准引用的杂项文件**（等待项目加载）
   - **否** → 分类为**带标准引用和语义错误的杂项文件**

### 退出选项

我们添加了一个退出标志，选项名称为 `dotnet.projects.enableFileBasedPrograms`。如果基于文件的程序体验出现问题，VS Code 用户应将对应设置 `"dotnet.projects.enableFileBasedPrograms": false` 以恢复到旧的杂项文件体验。

我们还有第二个更细粒度的退出标志 `dotnet.projects.enableFileBasedProgramsWhenAmbiguous`。此标志依赖于前一个标志（即当 `enableFileBasedPrograms` 为 `false` 时忽略它）。这用于仅在从单个文件本身不清楚是否应将其视为基于文件的程序的情况下退出。`.cs` 文件中存在 `#:` 或 `#!` 指令强烈表明该文件是基于文件的程序，即使当 `enableFileBasedProgramsWhenAmbiguous` 为 `false` 时，此类文件的编辑器功能仍将继续启用。

> [!NOTE]
> 第二个标志在短期内使用，在此期间我们正在制定所需的启发式和跨组件 API 集，以有效且令人满意地解决一个具有顶级语句但没有指令的文件是否是复杂工作区上下文中基于文件的程序的问题。

## 基于文件的应用的 LSP 处理

当 C# 文件添加 `#:` 或 `#!` 指令时，它成为基于文件的应用。
从概念上讲，发生的是：文件同时成为 C# 源文件和项目文件。

相反，当所有 `#:`/`#!` 指令被删除时，它停止作为项目文件，并回到仅作为 C# 源文件。在此方案中，我们将包含 `#:` 的文件视为基于文件的应用的"入口点文件"。

我们正在为基于文件的应用添加对 `#:include` 指令的支持，允许用户指向单个文件或 C# 源文件的 `*` 全局模式，或其他附加文件（内容、资源等），这些文件应包含在基于文件的应用项目中。这使基于文件的应用项目在工作区中的行为更像普通项目。特别是，我们可以遇到如下情况：
- `Util.cs`（普通源文件）
- `App1.cs`，包含 `#:include Util.cs` 的基于文件的应用入口点
- `App2.cs`，同样包含 `#:include Util.cs` 的基于文件的应用入口点
- `MyProject.csproj`，也包含 `<Compile Include="Util.cs" />`

因为所有这些项目只是作为项目添加到主机工作区，所以预计"活动项目上下文"和多目标感知快速信息等功能对所有这些项目都能"正常工作"。

我们做的一个关键假设是：基于文件的应用*入口点*不能是普通项目的成员。例如，你不能有以下情况：
- `Util.cs`（普通源文件）
- `App1.cs`，包含 `#:include Util.cs` 的基于文件的应用入口点
- `MyProject.csproj`，包含 `<Compile Include="App1.cs" />` **<-- 此部分被视为格式错误**

通常对普通项目中存在 `#:`/`#!` 指令会报错。根据加载顺序，此类文件可能也可能不会被检测为基于文件的应用入口点。

在这种情况下，我们希望用户执行以下两件事之一来解决问题：
1. 删除 `#:`/`#!` 指令。在这种情况下，我们将卸载基于文件的应用。
2. 将基于文件的应用入口点从任何普通项目中移除。

我们期望适当的项目系统能够观察到上述任一更改，并在用户纠正错误后将工作区移入"健康"状态。

### `FileBasedProgramsProjectSystem`

管理基于文件的程序和杂项文件的项目。

此项目系统在执行项目的设计时构建时，有效地执行[基于文件的应用检测](#file-based-app-detection)中描述的分类过程，并将项目状态转换为与最新分类匹配。

它使用基于文件的程序入口点文件，将其转换为虚拟 msbuild 项目，然后对该项目运行设计时构建。如果检测到缺少资产，它也可能还原虚拟项目。

它使用文件监视器监视项目全局模式，并对相关更改（例如 `#:` 指令的更改）重新执行设计时构建。

## 自动发现

Roslyn LSP 将自动发现并加载打开的工作区文件夹中的基于文件的应用。用户可以通过设置 `"dotnet.fileBasedApps.enableAutomaticDiscovery": false` 来退出此发现过程。
对于此功能在 VS Code C# 扩展中的首次发布，该设置在稳定发布渠道中默认禁用，在预发布渠道中默认启用。

工作区中的某些子文件夹被排除在此发现过程之外：
- 任何包含 `.csproj` 文件的文件夹。
- 名称按约定保留用于构建输出的任何文件夹，例如 `artifacts`、`bin` 和 `obj`。
- 在文件系统中标记为"隐藏"的任何文件夹。`.git` 和 `.vs` 通常属于此类。

在工作区中首次执行发现时，LSP 将读取打开的工作区文件夹中所有未被上述条件排除的 `.cs` 文件。如果文件内容以 `#!` 开头，则将其标记为基于文件的应用并加载。

每次发现通过后都会创建一个缓存文件并存储在用户临时目录中。此文件保存：
- 上次发现通过开始的时间。
- 上次发现通过中找到的基于文件的应用路径。
- 上次发现通过中发现包含 `.csproj` 文件的文件夹路径。

缓存数据允许在后续发现通过中进行以下优化：
- 允许不读取上次写入时间早于缓存时间的任何 C# 文件。
- 允许减少在上次写入时间早于缓存时间的目录中列出文件的次数。

### `#!` 要求

此设计要求文件以 `#!` 开头才能参与发现。
具体而言，可发现文件必须以字节序列 `0x23, 0x21`（ASCII/UTF-8 `#!`）或字节序列 `0xEF, 0xBB, 0xBF, 0x23, 0x21`（UTF-8 BOM 后跟 `#!`）开头。

原因是：我们预计将对非入口点文件添加 `#:` 支持。这意味着仅有 `#:` 不足以将文件识别为绝对的入口点。

因此，需要搜索 `#:` 和顶级语句至少两者。对于在编辑器中显式打开的文件，此成本是可以接受的，但对于广泛的发现通过来说有点高。

为此，我们打算将 `#!`-at-start 作为基于文件的应用入口点的标准。我们计划发布一个分析器，该分析器会对包含 `#:include` 和顶级语句但顶部没有 `#!` 的文件报告警告。

## 未来考虑

本节不旨在作为永久文档，而是作为我们在不久的将来可能在此领域进行的一系列更改的路线图。阅读/理解此内容对于评估当前正在审查的 PR 来说不是必要的。即当前 PR 实际实现的任何内容都在前面的部分中介绍了。

### 将没有指令的文件视为基于文件的应用入口点

**带标准引用和语义错误的杂项文件**是我们实际上为了避免还原我们不是 100% 确定是基于文件的应用的东西而设置的一个分类。这特别包括具有顶级语句但没有 `#:`/`#!` 指令的文件。

我们将来可能希望进行更改，停止将此分类用于磁盘上存在的文件，而是将不属于普通项目、包含顶级语句且没有 csproj 范围的文件分类为基于文件的应用。这将在某些情况下提高编辑器的准确性，并使避免显示*这是一个杂项文件，可能会出错*弹出窗口变得更容易。

### 允许非入口点文件包含 `#:` 指令

我们正在考虑将来添加对非入口点文件包含 `#:` 的支持。在这种情况下，我们需要额外的信息位来区分入口点和非入口点。
对于*多文件基于文件的应用*，用户应在入口点文件顶部使用 `#!`，以便于识别。
对于*单文件基于文件的应用*，我们认为仅使用 `#:` 和顶级语句在一起就足以将在编辑器中显式打开的文件识别为入口点。

### 在不进行完整解析的情况下检查顶级语句是否存在

目前有些情况下，我们可能需要对文件进行额外解析，只是为了检查它是否包含顶级语句。这通常是我们希望避免的情况，我们更希望使用以下模式之一：文件已经存在于某个项目中并具有我们可以增量检查的语法树，或者我们为执行不需要完整解析的启发式方法设计其他解决方案。
