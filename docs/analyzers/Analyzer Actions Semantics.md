**定义：**

 - **会话：** 对于批量编译，指单个编译。对于托管环境，指可能包含多个编译的任意持续时间。特别是对于 VS，会话与分析器引用的生命周期和范围共享，这可能对应于已加载的项目、已加载的解决方案或 VS 进程。
 - **分析器：** 从 `DiagnosticAnalyzer` 派生的类型的实例。
 - **操作：** 通过 \*\*\*Context 类实例的 Register\*\*\*Action 方法注册的方法实例，将根据情况自动应用。每个操作都与一个分析器关联，并具有一个类型（编译开始、语法树、符号、代码块开始、语法节点、代码块、代码块结束、语义模型、编译、编译结束之一）。
 - **操作块：** 包含操作节点的单个可执行代码单元，例如方法体、参数默认值、字段初始化器等。操作块主要用于使用 `IOperation` API 对可执行代码进行语义分析。
 - **代码块：** 包含语法节点的单个可执行代码单元，例如方法体、参数默认值、字段初始化器等。代码块主要用于使用语言特定的语法 API 对可执行代码进行语法和/或语义分析。但是，建议分析器使用操作块和 `IOperation` API 对可执行代码进行语义分析，以提高性能并实现更简单的、与语言无关的实现。
 - **符号成员：** 给定 `ISymbol` 的零个或多个成员符号列表。对于命名空间或类型符号，这些是 [INamespaceOrTypeSymbol.GetMembers](http://source.roslyn.io/#Microsoft.CodeAnalysis/Symbols/INamespaceOrTypeSymbol.cs,23) API 返回的成员，包括嵌套符号。方法、字段、事件和属性符号没有成员符号。

**公理：**
 - 假定两个不同的分析器不共享状态。除非它们在 `DiagnosticAnalyzer.Initialize` 方法中通过调用 `AnalysisContext.EnableConcurrentExecution` 显式配置并发执行，否则单个分析器的任何操作都不会并发执行。不同分析器的操作可能会并发执行。（随着我们开发更复杂的并发策略，这可能会得到改进，但操作通常不需要锁来避免竞态条件的情况将保持不变。）
 - 用于注册操作的上下文会影响注册的生命周期，但不会以其他方式影响应用操作的语义。
 - 如果分析器具有任何给定类型的多个适用操作，则所有这些操作将以任意顺序应用。

**操作的顺序：**

分析器的 `DiagnosticAnalyzer.Initialize` 方法在任何分析器操作可以应用之前被调用。`Initialize` 方法注册的操作构成每个编译使用的初始操作集。`Initialize` 方法可能在会话中的分析器实例上被多次调用，但保证对于任何给定的编译只调用一次。

以下与操作调用相关的规则适用于每个编译或使先前分析无效的任何环境变更（例如，对分析器选项的更改或对分析器使用的非编译文档的更改）。出于本规范的目的，此类环境变更的作用就像它们创建了一个新的编译。

除了由开始和结束操作（编译开始、编译结束、符号开始、符号结束、操作块开始、操作块结束、代码块开始和代码块结束操作）引入的约束外，任何操作的调用没有顺序约束。一种类型的操作的调用可能与另一种类型的操作的调用交错。给定的主机环境可能看起来提供了可预测的顺序——例如，在语义模型操作之前调用语法树操作，或在代码块操作之前调用符号操作，但依赖于这种顺序是不安全的。

 - 编译开始操作被调用一次，在分析器的任何其他类型的操作被调用之前。由编译开始操作注册的操作仅适用于该编译。
 - 语法树操作对每个源文档调用一次。可以期望语法树操作在文档解析后尽早被调用，但这不是保证的。
 - 符号操作在类型或类型成员声明的完整语义处理后调用一次，前提是该符号具有与注册操作时提供的类型之一匹配的类型。
 - 符号开始操作对于注册的 `SymbolKind` 的每个符号恰好调用一次，在分析器的任何其他类型的操作对符号或符号成员调用之前。符号开始操作可以注册以下操作：
    - 符号及其成员内的语法和操作节点的操作：操作操作、操作块操作、操作块开始操作、语法节点操作、代码块操作和代码块开始操作。
    - 符号的符号结束操作，在分析器的所有其他适用操作对符号及其成员调用之后。
 - 操作块开始操作对每个操作块调用一次，在任何适用于块内节点的操作节点操作之前。由操作块开始操作注册的操作仅适用于该操作块。
 - 操作操作对每个操作节点调用一次，前提是该操作节点具有与注册操作时提供的类型之一匹配的类型。
 - 操作块操作对每个操作块调用一次。
 - 操作块结束操作对每个操作块调用一次，在调用任何适用于块内节点的操作节点操作和块的任何操作块操作之后。
 - 代码块开始操作对每个代码块调用一次，在任何适用于块内节点的语法节点操作之前。由代码块开始操作注册的操作仅适用于该代码块。
 - 语法节点操作对每个语法节点调用一次，前提是该语法节点具有与注册操作时提供的类型之一匹配的类型。
 - 代码块操作对每个代码块调用一次。
 - 代码块结束操作对每个代码块调用一次，在调用任何适用于块内节点的语法节点操作和块的任何代码块操作之后。
 - 语义模型操作对每个源文档调用一次。
 - 编译操作被调用一次。
 - 编译结束操作在分析器的所有其他适用操作被调用之后调用一次。
 
总结顺序规则：
- 对于给定的编译，编译开始操作首先运行，编译结束操作最后运行。
- 对于给定的符号，符号开始操作在符号及其成员的操作/语法/符号操作之前运行，这些操作/语法/符号操作在符号结束操作之前运行。
- 对于给定的操作块，操作块开始操作在操作节点操作之前运行，操作节点操作在操作块结束操作之前运行。
- 对于给定的代码块，代码块开始操作在语法节点操作之前运行，语法节点操作在代码块结束操作之前运行。
- 否则，操作调用没有顺序保证。

**实现自由度：**

如果实现确定给定操作在会话中的两个编译中将具有等效效果，则它可以省略为第二个编译调用该操作，而是保留为第一个编译调用它的效果。例如，在 IDE 中，对源文档的编辑不一定会导致对其他文档中定义的无关符号等调用操作。

主机环境可以任意且无限期地延迟操作的调用。例如，IDE 可能会延迟调用某些操作，直到编辑活动明显暂停，在此期间可能会启动多个编译。某些操作可能根本不会为临时编译执行，因为当下一个编译开始时它们将被取消。在这种情况下，对于给定的编译，编译开始操作可能在没有相应的编译结束操作运行的情况下运行，同样，代码块开始操作可以在没有相应的代码块结束操作运行的情况下运行。因此，不能依赖结束操作来释放由开始操作获取的资源。
