# 开发语言功能

向 C# 或 VB 添加新功能是一项非常严肃的工作，即使是最（看似）简单的功能，通常也需要多次迭代才能完成。这是因为语言变更本身固有的复杂性，以及需要考虑新功能在 Roslyn 代码库各层（IDE、调试、脚本等）中的影响。因此，语言工作在单独的分支中进行，直到功能达到可以合并到主编译器的状态。

本文介绍语言功能*实现*被考虑、原型化并完全接受进入语言的流程。该流程供编译器团队和社区使用。

## 流程

1. **提交功能规范**：规范草案应以 GitHub issue 的形式提交，并包含：

        * 功能描述（包括涉及的语法变更）
        * 受影响领域的讨论，如重载解析和类型推断。在确定潜在影响时，仔细考虑语言规范的主要领域
        * API 接口变更建议。

    功能规范草案与语言设计讨论不同。讨论是非常开放性的，通常针对那些根本无法在语言中实现的功能。功能规范草案是按照提案实施该功能的意向声明。在许多情况下，这将从语言设计讨论中产生。

    强烈建议链接到所提议功能的早期原型。这让语言团队能清楚理解功能的设计意图，通常可以作为功能分支的初始提交。

    *注*：目标不是有一个 100% 完整的设计，而是对语言功能预期如何工作有一个大致了解。

1. **讨论**：讨论的目的是帮助澄清设计，并确定语言团队是否希望批准该提案。随着讨论的进行，建议作者更新规范草案以纳入不断变化的设计。这样，后来加入的人可以通过阅读帖子顶部来了解最新提案。

1. **语言团队做出决定**：当语言团队认为有足够信息时，他们将相应地更新 issue：

    * **准备好进行原型开发**：该功能值得作为语言原型继续推进。工作进入"原型"阶段。
    * **无法处理**：讨论未能形成团队可以处理的设计。
    * **按提案拒绝**：总体来说该提案是个好主意，但提议的实现细节不够高效。在大多数情况下，语言团队将尝试将讨论引导到更简洁的设计，但理解有时重新开一个新 issue 会更简单。
    * **积压**：该功能是个好主意，有可接受的设计，但语言团队目前无法分配资源进行此工作。
    * **不会修复**：该功能不是语言团队感兴趣的发展方向。

    当社区驱动的功能被认定为"准备好进行原型开发"时，语言团队成员将与之配对。该团队成员将作为与社区沟通的主要联系人，并帮助推动功能完成。

1. **原型阶段**：所有功能最初都在单独的分支中进行原型开发。分支名为 `features/<功能名称>`。当功能规范草案被标记为"准备好进行原型开发"、语言团队成员担任功能负责人，以及[分配开发者](#community-contributions)实现该功能时，这些分支将被创建。开发者负责在整个开发过程中保持 `docs/features/<功能名称>.md` 中的规范草案处于最新状态。

     开始新功能工作时，使用 `new-compiler-feature` Copilot 技能（在 `.github/skills/new-compiler-feature/` 中）来设置跟踪：它创建测试计划 issue、`Feature - <名称>` 标签，更新[语言功能状态](../Language%20Feature%20Status.md)页面，并链接相关 PR。从实现（部分）功能的 PR 运行它。

     所有功能在完全接受进入语言之前应默认关闭。新功能只应在编译器选项 `/features:<功能名称>` 显式包含时才启用。这有助于确保无论当前分支中有哪些原型，语言始终处于可发布状态。

1. **原型决定**：当原型阶段结束时，语言团队将对功能做出决定：
    * **准备集成**：功能实现没有发现阻塞性问题，拥有健全的测试套件，并且在编译器、IDE、调试器等方面足够完整，可以集成到所选分支的主树中。测试套件需要同时关注功能的独立性以及与语言其他领域的组合。
    * **迭代**：原型的经验产生了有价值的反馈，建议修改规范草案。工作回到讨论状态，这可能导致规范草案和原型的变更（甚至放弃该功能！）。这通常发生在原始规范草案留下一些语言细节未指定，需要根据实现经验解决，或者原型经验建议重新考虑某些语言设计决策时。
    * **关闭**：功能遇到了不可预见的设计问题、实现障碍等，导致被移除。

1. **完善**：一旦功能集成到主树中，就是时候锁定和完成实现了。这包括原型中未完成的任何剩余领域，以及完成所有边角情况和测试（大量测试！）。

    主树中但不与 langversion 开关绑定的功能称为实验性功能。它们随时可以被撤回或更改。

1. **接受**：当语言团队认为功能完整时，它将被接受进入语言。相应的功能标志将被移除，功能将与适当的 `/langversion` 选项绑定。

重要的是要注意，**没有任何保证**一个功能会成为语言的一部分，直到它达到"接受"状态**并且**在官方构建中发布。处于任何其他状态的功能——无论多么完整——如果语言团队认为有必要，都可以在最后时刻被撤回。

在原型阶段完成后撤回功能不是语言团队轻易做出的决定；然而，在过去，不可预见的情况迫使我们这样做。最好的例子是决定在 C# 6.0 时间表的很晚阶段撤回无参数结构构造函数。

## 社区贡献

本文的目的是整理语言团队实现语言功能的流程，以便社区可以贡献实现。社区成员可以为正在进行的原型做出贡献，在某些情况下也可以成为原型的主要推动者。

希望推动特定功能的社区成员可以通过在已接受的功能规范草案上发表评论来提出请求。即使由社区成员驱动，语言原型也会占用语言团队不少时间。因此，我们可能只会将此请求授予在较小领域积极贡献项目的社区成员。

## 常见问题

- **功能分支可以被变基吗？**：这由功能开发者自行决定。一旦功能合并到主分支（main、feature）中，就不再允许变基。

- **为什么我的语言功能 PR 被关闭了？**：语言功能不会作为直接 PR 被接受到主分支。即使是简单的功能也太复杂，高级语言团队成员无法以这种方式完成。功能需要遵循上述流程才能被纳入语言。

- **为什么功能默认禁用？**：语言功能在我们的发布产品所在的同一分支（通常是 main）中花费相当长的时间。Roslyn 的 main 分支始终保持可发布状态。为了支持这一点，所有尚未完全接受进入语言的功能必须被禁用。

- **功能分支和 main 分支有什么区别？**：main 分支代表将在当前版本编译器更新中发布的代码。功能分支代表仍在进行中的功能，因此尚未合并。

- **为什么早期功能没有提交规范草案？**：元组和局部函数等功能是在此流程完全开发之前完成的。它们的规范草案在内部讨论，最终成为 `docs/feature/<功能名称.md>` 文件。

- **为什么创建功能分支的门槛这么高？**：语言功能，即使由社区驱动，也需要语言团队进行大量的长期资源投入。团队必须定期进行代码审查、提供指导、帮助解决架构问题、贡献代码等...
