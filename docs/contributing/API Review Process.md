# API 审查流程

.NET 有着极其重视 API 可用性的悠久历史。因此，我们通常会审查添加到产品中的每一个 API。本页讨论我们如何为 Roslyn 组件进行设计审查。

## 哪些 API 应该被审查？

经验法则是，我们（**dotnet/roslyn**）审查添加到此仓库的 CodeAnalysis（编译器）、Workspaces、Features 和 EditorFeatures 库中的每一个公共 API。LanguageServices dll 中的 API 通常不会被审查，因为它们不打算供一般使用；它们供 Visual Studio 使用，我们对这些 API 没有兼容性要求。

## 步骤

1. **请求者提交问题**。问题描述应包含一个代表新 API 草图的规范说明，包括如何使用 API 的示例。目标不是获得完整的 API 列表，而是对新 API 大致外观以及在哪些场景中使用有一个良好的把握。请使用[此模板](https://github.com/dotnet/roslyn/issues/new?template=api-suggestion.md)。问题应该有标签 `Feature Request` 和 `Concept-API`。[这里](https://github.com/dotnet/roslyn/issues/53410)是一个强 API 提案的好例子。

2. **我们分配一个负责人**。我们将从我们这边分配一个专门的负责人来赞助这个问题。这通常是 API 提案或设计更改请求提交的[区域负责人](../area-owners.md#areas)。

3. **讨论**。讨论的目标是帮助分配者决定我们是否要推进提案。在此阶段，目标不一定是进行深入审查；相反，我们希望确保提案是可操作的，即有具体的设计、API 草图和一些展示如何使用的代码示例。如果需要更改，负责人将设置标签 `api-needs-work`。要进行更改，请求者应编辑最顶层的问题描述。这允许稍后加入的人理解最新的提案。为避免混淆，请求者可以维护一个小的更改日志，如加粗的"更新："后跟更改的项目符号列表。当反馈得到解决时，请求者应通知负责人重新审查更改。

4. **负责人做出决定**。当负责人认为有足够的信息来做出决定时，他们将相应地更新问题：

    * **标记为审查**。如果负责人认为提案是可操作的，他们将用标签 `api-ready-for-review` 标记问题。
    * **关闭为不可操作**。如果问题没有获得足够的关注以提炼成具体的提案，负责人将关闭问题。
    * **关闭为按提议不会修复**。有时，提出的问题是好的，但负责人认为具体的提案不是解决问题的正确方法。在大多数情况下，负责人会尝试将讨论引导到我们认为合适的设计方向。然而，对于某些提案，问题在于设计的核心，不容易在不开始新提案的情况下更改。在这些情况下，负责人将关闭问题并解释设计存在的问题。
    * **关闭为不会修复**。同样，如果提案将产品带向我们根本不想去的方向，问题也可能被关闭。在这种情况下，问题不在于提议的设计，而在于问题本身。

5. **API 得到审查**。进行审查的组称为 *RAR*，代表 *Roslyn API 审查者*。在审查中，我们会做笔记并提供反馈。审查后，我们将在 [API Review 仓库](https://github.com/dotnet/apireviews)和相关问题的末尾发布笔记。可能有多种结果：

    * **批准**。在这种情况下，标签 `api-ready-for-review` 被替换为 `api-approved`。
    * **需要工作**。如果我们认为提案还没有准备好，我们将把标签 `api-ready-for-review` 替换为 `api-needs-work`。
    * **拒绝**。如果我们认为提案不是我们想要追求的方向，我们将简单地写一条评论并关闭问题。

## 审查日程

有三种方法可以获得 API 审查（本节适用于 API 区域负责人，但为了流程的透明度而包含在内）：

* **进入待办事项列表**。一般来说，在 `dotnet/roslyn` 中提交问题并在其上应用标签 `api-ready-for-review` 将使您的问题在 API 审查期间显示。缺点是我们通常从最旧到最新遍历待办事项列表，所以您的问题可能暂时不会被查看。问题的进度可以通过 <https://apireview.net/backlog?g=Roslyn> 跟踪。
* **快速通道**。如果您需要绕过待办事项列表，请同时应用 `api-ready-for-review` 和 `blocking`。所有阻塞问题都会在我们遍历待办事项列表之前被查看。
* **专门审查**。如果您作为区域负责人的问题需要一个小时或更长时间，请发送电子邮件到 roslynapiowners，我们会预订专门的时间。我们还为语言功能 API 预订专门时间：作为新语言功能一部分添加的任何 API 都需要有专门的审查会议。经验法则：如果 API 提案有超过十几个 API，API 有复杂的策略，或者 API 是功能分支的一部分，那么您需要 60 分钟或更长时间。如有疑问，请发送邮件到 roslynapiowners。

API 审查委员会目前每 2 周开一次会。

## 拉取请求

针对 **dotnet/roslyn** 添加新公共 API 的拉取请求在获得批准之前不应提交。此外，我们不想收到进行中的工作 (WIP) PR。原因是我们希望减少待处理 PR 的数量，以便我们可以专注于社区期望我们采取行动的工作。

如果您想与其他人合作设计，请随意在您自己 fork 的分支中进行工作。如果您想在 PR 的描述中跟踪您的待办事项，您可以随时针对您自己的 fork 提交 PR。此外，请随意通过从您在上面第一步中针对 **dotnet/roslyn** 提交的问题链接您的 PR 来宣传它。

对于 **dotnet/roslyn** 团队成员，允许在完整的 API 审查会议之前提交和合并 PR，但您应该获得区域负责人的批准，并且预计 API 将在正式审查会议中被涵盖后才能发布。对于此类问题，请与 API 区域负责人合作，确保问题被适当标记为阻塞，以便在短期内得到涵盖。
